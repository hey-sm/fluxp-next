## Upload SDK 

1. 项目背景  
我做过一个从 0 到 1 的 Upload SDK，主要负责前端侧的设计和落地。项目背景是业务里的图片、视频、普通文件、大文件上传实现比较分散，不同页面各自封装，上传协议、组件能力和异常处理都不统一，导致重复开发较多，接入和维护成本也比较高，所以我把上传能力沉淀成了一套统一的 SDK。

2. 整体架构  
整体上我把 SDK 拆成四层。第一层是协议层，和后端统一上传初始化、分片上传、查询上传状态、合并分片等接口规范；第二层是上传引擎层，负责切片、hash 计算、并发调度、失败重试、暂停恢复和进度汇总；第三层是能力封装层，对外提供统一的 API 和 Hooks；第四层是组件层，封装上传按钮、拖拽上传、图片上传、视频上传等通用组件。这样业务方只需要调用统一 API，不需要关心底层到底是普通直传还是分片上传。

3. 核心实现  
在实现上，我主要做了三件事。第一，利用策略模式和工厂模式统一上传策略，把小文件直传、大文件分片、图片上传、视频上传抽象成不同策略，业务接入时只需要传配置，SDK 自动选择对应方案。第二，把上传前校验、初始化任务、执行上传、完成回调这些公共流程抽到基类里，不同上传器只处理自己的差异逻辑，保证 SDK 和业务解耦。第三，针对大文件上传实现了任务调度器，对分片上传做并发控制，而不是直接 `Promise.all`，这样可以避免请求打满，也支持单分片失败重试，不需要整文件重传。

4. 文件唯一性与断点续传  
文件唯一性这块，前端会先计算文件 hash，大文件场景下放到 `Web Worker` 中异步处理，避免阻塞主线程。初始化上传时把 hash、文件信息和业务标识传给后端，后端基于这些信息生成稳定的存储 key，这样可以支持秒传、去重和断点续传，也保证文件存储路径唯一且可控。前端则基于 `uploadId + fileHash` 管理上传任务，通过查询已上传分片，只补传缺失部分。

5. 接入方式  
在接入层面，我没有让业务页面直接处理复杂上传逻辑，而是通过 Hooks 和组件统一收口。页面层只负责选文件、展示进度和处理结果，真正的校验、任务状态、上传生命周期和异常处理都放到 SDK 内部管理，这样业务代码更轻，也方便后续扩展图片压缩、视频封面提取、上传埋点等能力。

6. 项目难点  
我觉得这个项目主要有三个难点。第一是大文件上传的稳定性，需要平衡切片大小、并发数、失败重试和整体吞吐；第二是断点续传，要处理页面刷新、重复上传和已上传分片校验；第三是通用性，既要保证能力统一，又要保留足够的业务扩展空间，所以设计时我保留了清晰的配置项和生命周期回调。

7. 项目结果  
最终这个 Upload SDK 把原来页面级零散的上传实现沉淀成了一套可复用的基础能力，统一了上传方式和接入标准，也降低了业务侧开发和维护成本。对于图片、视频和大文件上传场景，整体稳定性和可维护性都有比较明显的提升。

## 首屏优化

1. 先做数据采集，不靠主观判断

   收到客户反馈首屏慢之后，我先补了性能埋点，重点采集 `FP`、`FCP`、`LCP`、`CLS`、`TTFB`，以及首屏接口耗时、静态资源加载耗时和页面 waterfall。这样可以先判断瓶颈到底在网络、服务端、资源体积还是主线程执行，而不是一上来就盲目优化。

2. 再做问题定位和复现

   我会先看线上埋点数据，按页面、网络环境、地区和浏览器维度聚合，确认是不是普遍问题，以及慢主要集中在哪些用户场景。然后在本地用 Chrome DevTools、Lighthouse 和弱网模拟做复现，重点看 LCP 元素是谁、资源请求有没有排队、JS 包是不是过大、首屏接口和静态资源是不是互相抢占带宽。

3. 核心问题通常拆成两类

   第一类是网络链路问题，比如在 `HTTP/1.1` 下请求并发受限，静态资源多的时候容易出现队头阻塞，关键资源被排队；第二类是前端代码问题，比如首屏包体积过大、重复代码多、非首屏模块提前加载，导致下载和执行都变慢。

4. 网络层的优化思路

   针对网络问题，我会优先推动静态资源链路升级到支持多路复用的协议，比如 `HTTP/2`，缓解 `HTTP/1.1` 下的请求排队问题。同时检查 CDN、网关和静态资源压缩配置，确认 `gzip` 或 `br` 是否生效，并尽量减少冗余请求头、Cookie 和非必要请求。对于首屏资源，还会调整加载优先级，让真正影响首屏渲染的资源优先返回。

5. 代码层的优化思路

   针对代码问题，我一般会做几件事：一是提取公共依赖和重复代码，减少重复打包；二是通过动态 `import` 把非首屏模块延后加载；三是对重型组件和第三方库做懒加载或替换；四是对首屏图片、字体和静态资源做压缩与格式优化。核心目标就是减少首屏下载体积和主线程执行压力。

6. 为什么要做自动代码提重

   因为很多页面在长期迭代后，容易出现多个模块重复依赖同一个工具库、重复实现同类逻辑，最后导致产物里出现重复代码。我在优化时会结合 bundle 分析工具去看哪些模块重复进入多个 chunk，再通过公共 chunk 抽离、工具函数收口、按需引入等方式减少重复打包，从而直接压缩首屏资源体积。

7. 优化后怎么验证结果

   优化完成后，我不会只看本地感觉，而是通过三层验证：先看本地 Lighthouse 和 Performance 面板，再看预发环境的 waterfall 和包体积变化，最后对比线上真实用户监控数据，比如 `P75 LCP`、首屏成功率、资源加载耗时。只有线上数据确认下降，才能说明优化真正生效。

8. 如果面试官问“你这次为什么能把 LCP 降低 80%”

   我会回答，因为这次不是只做单点优化，而是先通过埋点和实验定位出主要瓶颈，再同时从网络协议、资源加载顺序、包体积治理和代码执行成本几个方向一起优化。最终关键首屏资源的下载和渲染链路明显缩短，所以 `LCP` 才会有比较明显的下降。

## 1. 请简述一下 SSE 和 WebSocket 的区别，在 AI 流式输出场景下你会如何选择？

SSE 是基于 HTTP 的单向推送，服务端可以持续向前端推消息，前端通过 `EventSource` 或基于 `fetch + ReadableStream` 读取流式内容；WebSocket 是全双工通信，前后端都可以随时主动发消息。  

如果是 AI 对话这种“前端发一次请求，服务端持续返回 token”的场景，我通常优先选 SSE 或基于 HTTP 的流式响应。原因是它更贴近请求-响应模型，接入简单，天然适合文本流输出，也更容易和现有网关、鉴权、日志体系集成。  

如果场景变成多角色实时协同、客户端要频繁主动上报状态、需要服务端主动插入更多事件流，比如多人共享会话、工具执行状态广播、直播间 AI 助手联动，这种更强实时双向通信的场景，我才会优先考虑 WebSocket。

## 2. 在处理 AI 响应的长列表消息时，如何实现自动滚动到底部，并保证用户手动向上滚动时不被干扰？

核心思路是“只在用户本来就接近底部时自动滚动，否则尊重用户当前阅读位置”。  

具体做法一般是监听滚动容器的 `scrollTop`、`scrollHeight` 和 `clientHeight`，当满足“距离底部小于某个阈值”时，认为用户处于跟随模式，AI 新消息到来后自动滚动到底部；如果用户手动向上滚动超过阈值，就切换为非跟随模式，此时即使有新消息也不强制滚动。  

实践里我通常会配合一个底部锚点元素，用 `scrollIntoView` 或直接设置容器 `scrollTop`。同时为了减少高频更新导致的抖动，会把滚动操作放到 `requestAnimationFrame` 或批量更新后统一执行。

## 3. 请手写一个具有并发限制的请求调度器，模拟同时调用多个 AI 接口的场景。

可以把它抽象成一个并发池，核心是维护一个运行中计数和等待队列：

```ts
function createScheduler(limit: number) {
  let activeCount = 0
  const queue: Array<() => void> = []

  const next = () => {
    if (activeCount >= limit || queue.length === 0) {
      return
    }

    const run = queue.shift()
    run?.()
  }

  const add = <T>(task: () => Promise<T>) => {
    return new Promise<T>((resolve, reject) => {
      const execute = async () => {
        activeCount += 1

        try {
          const result = await task()
          resolve(result)
        } catch (error) {
          reject(error)
        } finally {
          activeCount -= 1
          next()
        }
      }

      queue.push(execute)
      next()
    })
  }

  return { add }
}

const scheduler = createScheduler(2)

const tasks = [1, 2, 3, 4, 5].map((id) =>
  scheduler.add(async () => {
    console.log(`start ${id}`)
    await new Promise((resolve) => setTimeout(resolve, 1000))
    console.log(`done ${id}`)
    return `result-${id}`
  }),
)

Promise.allSettled(tasks).then(console.log)
```

如果放到 AI 接口场景里，这种调度器可以用来控制多个模型请求、重试任务、分片任务或者工具调用，避免瞬时请求过多把浏览器和服务端都打满。

## 4. 谈谈你对 CSS 容器查询(Container Queries)的理解，它在 AI 侧边栏组件中有什么作用？

容器查询和传统媒体查询的区别在于，媒体查询看的是视口尺寸，容器查询看的是组件所在容器的尺寸。  

这在 AI 侧边栏场景里特别有价值，因为侧边栏可能出现在桌面端右侧、可折叠抽屉、嵌入式面板甚至主内容区下方，同一个组件在不同容器宽度下应该有不同布局，但不一定和整个窗口宽度直接绑定。  

比如当 AI 侧边栏容器宽度大于 480px 时，可以展示完整消息操作栏、推荐问题列表和工具状态；容器变窄时，就改成紧凑布局、折叠按钮文案、隐藏次要信息。这样组件会更独立，也更适合复用到不同宿主页面中。

## 5. 如何实现一个简单的 Markdown 实时渲染引擎，并处理其中的代码高亮？

最简单的实现思路是分两步：第一步把 Markdown 文本解析成语法树或者 HTML；第二步在渲染代码块时做语法高亮。  

如果是 React 场景，我通常会选择成熟链路，比如 `remark` 负责 Markdown 解析，`rehype` 负责 HTML 或 AST 处理，再配合 `shiki`、`highlight.js` 或 `rehype-pretty-code` 做代码高亮。  

实时渲染场景下还要注意两点：一是流式输出过程中内容是不完整的，不能每来一个字符就整篇全文重解析，可以做分段解析或者节流渲染；二是要做 XSS 安全处理，避免直接注入不可信 HTML。  

如果只是做一个简单版本，也可以先把普通文本和代码块拆开处理，普通文本走 Markdown 解析，代码块识别 ``` fence 后单独渲染并高亮。

## 6. 解释一下 JS 中的事件循环(Event Loop)，并说明宏任务与微任务在处理 AI 消息推送时的先后顺序。

事件循环可以理解成 JS 运行时调度任务的机制。同步代码先执行，执行栈清空后，会先处理当前轮次产生的所有微任务，再进入下一个宏任务。  

常见宏任务有 `setTimeout`、`setInterval`、I/O、消息事件；常见微任务有 `Promise.then`、`queueMicrotask`、`MutationObserver`。  

在 AI 消息推送场景里，比如我们读取流式响应后把内容塞进状态，很多框架内部状态更新和 `Promise` 回调会进入微任务队列，所以它们通常会先于下一轮 `setTimeout` 执行。也就是说，顺序一般是：当前同步逻辑执行完 -> 清空微任务 -> 再处理下一个宏任务。  

这点对前端更新节奏很重要，因为如果每次 token 到来都立即触发状态更新，微任务堆积过多也会让渲染变得很频繁，所以实际开发中常常需要做节流或批量刷新。

## 7. AI 流式输出时，前端如何解决“打字机”效果带来的频繁 DOM 渲染导致的性能问题？

核心思路不是“来一个 token 渲染一次”，而是“做缓冲区 + 批量提交”。  

具体做法通常是先把流式返回的 token 暂存到内存 buffer 里，再通过 `requestAnimationFrame`、定时节流或者最小批量长度控制，把多个 token 合并成一次状态更新。这样可以把原本几十上百次渲染压缩成每帧一次或者每几十毫秒一次。  

如果内容特别长，我还会把消息列表和单条消息拆开，避免整个聊天列表跟着重新渲染；对于 Markdown 实时解析，也可以延迟到一段内容稳定后再做，避免频繁全文重算。

## 8. 在开发 AI Agent 时，前端如何管理复杂的 Prompt 模板以及变量替换逻辑？

我的做法一般是把 Prompt 当成一种“可配置资源”，而不是散落在组件里的字符串拼接。  

通常会做三层抽象：第一层是模板定义层，负责维护系统 Prompt、工具 Prompt、角色 Prompt 等模板；第二层是变量注入层，负责把用户输入、上下文、业务参数、历史消息等按规则替换到模板里；第三层是校验和预览层，确保必填变量完整，并且在调试环境下能看到最终拼接结果。  

如果 Prompt 很复杂，我会把模板拆成结构化片段，比如指令区、上下文区、约束区、输出格式区，再统一组合。这样既方便复用，也方便后续做版本控制、A/B 测试和灰度发布。

## 9. 谈谈 Function Calling(函数调用)的前端实现流程，前端需要做哪些配合？

Function Calling 的核心是模型不直接返回最终自然语言，而是先返回“它想调用哪个函数、参数是什么”。  

前端通常要配合三件事。第一，负责展示函数调用中的状态，比如“正在查询天气”“正在搜索资料”；第二，处理多轮消息编排，也就是模型发出 tool call 后，把函数执行结果再回传给模型继续生成最终答案；第三，做好可视化和容错，比如参数校验失败、函数超时、函数返回为空时要如何提示用户。  

如果函数真正执行发生在后端，前端更多承担的是状态编排和 UI 展示；如果是前端工具能力，比如复制、下载、本地文件选择、打开弹窗等，前端就还要负责安全控制和权限判断。

## 10. 如何处理 AI 响应过程中的网络中断或内容截断？请设计一套容错机制。

我通常会把它拆成“检测、恢复、兜底”三层。  

检测层：识别请求超时、连接断开、流提前结束、消息结构不完整等异常。  
恢复层：如果支持幂等和断点续传，可以基于 `messageId`、`conversationId`、`cursor` 或已接收内容长度发起重试，尽量从中断位置继续；如果不支持真正续传，也至少要保留已生成内容，并提示用户重试。  
兜底层：把当前状态标记为“已中断”，展示明显但不打断阅读的提示，比如“回答可能未完整生成，可继续重试”，同时提供重新生成、继续生成、复制已有内容等操作。  

如果是生产场景，我还会补充超时重试次数、指数退避、用户主动取消和日志上报，方便排查到底是前端网络、网关中断还是模型服务超时。

## 11. 在直播间这种高频交互环境下，AI 组件如何与直播间原有的状态(如点赞、弹幕)进行高效同步？

这种场景的关键不是“把所有状态都塞给 AI 组件”，而是做状态分层。  

直播间原有的点赞、弹幕、在线人数这类高频状态，通常应该继续由直播间主状态中心统一维护；AI 组件只订阅自己真正需要的那部分派生数据，比如最新几条弹幕摘要、当前直播主题、用户身份信息，而不是直接订阅全部原始流。  

实现上我会用事件总线、状态管理库或者 selector 机制做精细订阅，避免高频状态更新把 AI 面板整体拖着重渲染。必要时还会对弹幕流做聚合和节流，比如每秒合并一次摘要，再同步给 AI 侧边栏。

## 12. 谈谈 React Fiber 架构对 AI 聊天机器人这类长文本更新场景的意义。

React Fiber 的意义在于它把原来不可中断的渲染过程拆成了可调度、可中断的工作单元，让 React 可以更细粒度地安排更新优先级。  

在 AI 聊天这种场景里，消息是持续增长的，内容可能很长，如果每次更新都同步阻塞整个渲染，会明显影响输入框、滚动和页面交互体验。Fiber 让 React 有机会把高优先级的用户交互和低优先级的大段文本渲染区分开，整体体验会更平滑。  

从开发者角度看，这也是为什么在 AI 长文本场景里，我们更要注意拆分组件边界、减少不必要渲染，并在合适的时候利用 React 的并发更新能力。

## 13. 如果 AI 返回的内容包含敏感信息被后端拦截，前端应该如何优雅地展示和处理？

我会优先把它当成一个“有结果但不可直接展示”的状态，而不是普通报错。  

前端展示上可以保留这次问答记录，但把回答区域替换成合规提示，比如“该内容因安全策略未完整展示”，同时提供重新提问、修改问题、联系客服或查看规范说明等操作。这样既不会让用户觉得系统坏了，也能清楚表达拦截原因。  

技术上要注意区分敏感拦截、网络失败、模型失败这些不同错误类型，避免全部展示成同一种 toast；如果后端会返回脱敏片段或风险类型，前端也可以针对性展示更友好的解释。

## 14. 请解释 WebWorker 在 AI 前端开发中的应用场景，你会用它处理什么？

WebWorker 的核心价值是把高计算量、非 UI 任务放到主线程之外执行，避免页面卡顿。  

在 AI 前端开发里，我会优先把这些任务放进 Worker：Markdown 或富文本的大段解析、代码高亮、文件 hash 计算、向量数据预处理、长文本分块、聊天记录导出、复杂 JSON 结构格式化，或者一些本地推理前后的数据预处理。  

如果是 AI 聊天产品，最典型的使用场景就是大文件上传前的 hash 计算、长消息的 Markdown 解析和本地知识库切片预处理。原则上只要任务计算重、耗时长、又不需要直接操作 DOM，就很适合考虑放到 Worker 里。
