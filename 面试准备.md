## Monorepo 是怎么落地的？

1. 通过 pnpm-workspace.yaml 管理多个应用和公共包，将内部系统、供应商平台以及通用能力模块统一收敛到同一代码仓库，解决多项目间公共代码分散、版本不同步和重复维护的问题。
2. 通过 packages/ui、packages/hooks、packages/utils、packages/types、packages/configs 的拆分方式沉淀通用组件、组合式逻辑、工具方法、类型定义和工程配置，让多个业务应用直接复用同一套基础能力。
3. 通过 turbo.json 统一编排 dev、build、lint、type-check 等任务，并配置 dependsOn 和 outputs，利用本地缓存/增量执行减少未变更模块的重复构建，提升开发和 CI 执行效率。
4. 通过统一 TypeScript path、ESLint、Prettier、提交规范和环境变量管理，收敛不同项目的工程配置，降低多人协作时的接入成本和维护复杂度。
5. 通过“先抽配置、再抽工具、再抽组件、最后抽业务通用模块”的渐进式迁移方式完成旧项目升级，避免一次性重构带来的风险，保证业务迭代不中断。

## 审批流程权限控制是怎么做的？

1. 通过 权限码 + 审批状态 + 节点角色 + 操作映射表 建立前端权限判定模型，将提交、撤回、审批、驳回、转交、作废等操作的可见性和可操作性从页面逻辑中抽离出来统一管理。
2. 通过维护 actionMap 或 permissionMap 配置表，将每个操作对应的 permissionCode、适用状态、适用角色、前置条件、禁用原因 统一配置，页面只根据当前单据状态和用户权限动态渲染按钮，避免在多个页面中写大量 if/else。
3. 通过封装 useApprovalPermission 这类 hooks，统一收口当前用户权限集、流程节点信息、单据状态和按钮操作规则，输出 visibleActions、disabledActions、actionReason 等结果，降低页面层判断复杂度。
4. 通过“显示权限”和“执行权限”分层控制页面行为，前端负责按钮显隐、禁用态和文案提示，提交操作前再结合接口返回做二次校验，避免因前端缓存状态导致误操作。
5. 通过详情页容器组件统一承载头部操作区、流程区、表单区和审批记录区，并将操作按钮渲染逻辑下沉到公共容器内部，不同审批单据只需要传入状态字段、权限配置和业务表单即可复用整套交互。
