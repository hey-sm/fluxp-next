# TypeScript 类型推导详解

很多人把 TypeScript 理解成“所有地方都要手写类型”，但真实开发中，TypeScript 的体验很大程度上恰恰来自类型推导。你不需要给每个局部变量都手动标注，编译器会根据赋值、上下文、返回值和控制流分析推导出相当精确的类型。

理解类型推导非常重要，因为这会直接决定：

- 什么地方应该让编译器自动推
- 什么地方应该显式标注
- 为什么某些变量会被推成过宽或过窄的类型
- `as const`、字面量收窄、上下文类型为什么会影响后续 API 设计

官方文档主线参考：

- [Type Inference](https://www.typescriptlang.org/docs/handbook/type-inference.html)
- [Literal Inference](https://www.typescriptlang.org/docs/handbook/2/everyday-types.html#literal-inference)

## 类型推导到底在做什么

一句话概括：TypeScript 会根据你写出的代码上下文，自动猜测某个值最合理的类型。

它解决什么问题：如果每个局部变量、回调参数和函数返回值都要手写类型，开发成本会非常高；而如果编译器完全不推导，类型系统也会失去可用性。

所以 TypeScript 的真实工作方式通常是：

- 能从代码明确得出的，尽量自动推导
- 无法安全得出的，再要求开发者补充标注

## 变量推导

最基础的推导就是变量初始化推导。

```ts
const name = 'Ada'
const count = 1
const visible = true
```

这里 TypeScript 会根据初始值推导出类型，而不是要求你全部写成：

```ts
const name: string = 'Ada'
```

这种推导的价值在于：

- 保持代码简洁
- 避免重复信息
- 让局部变量自然携带类型信息

但这只是最基础的一层，更重要的是接下来的“推得多精确”。

## 宽化与收窄

TypeScript 不只是推导类型，还会决定这个类型要不要“宽化”。

例如：

```ts
let status = 'idle'
```

这里通常会被宽化为 `string`，因为 `let` 变量后续还可能改成其他字符串值。

而：

```ts
const status = 'idle'
```

则更可能保持字面量类型 `'idle'`，因为它不会再被重新赋值。

这说明一个重要原则：

- `const` 更容易保留精确字面量信息
- `let` 更容易被推成更宽泛类型

这也是为什么很多状态常量、配置对象、路由名、动作类型会特别受益于 `const` 与字面量推导。

## 对象字面量为什么常常被推得比你想的更宽

看一个例子：

```ts
const config = {
  theme: 'dark',
}
```

很多人会以为 `config.theme` 会被推成 `'dark'`，但默认情况下它往往被推成 `string`。这是因为对象属性默认仍然可能被修改，编译器会做更宽的推断。

这会直接影响很多场景，例如：

- 配置对象
- Redux / Zustand / Pinia action type
- 路由定义
- 组件变体映射

如果你确实需要保留精确字面量信息，就要进一步理解 `as const`。

## as const

一句话定义：`as const` 会让 TypeScript 尽量保留值的最精确只读字面量类型。

它解决什么问题：默认对象和数组推导常常过宽，而很多配置型数据恰恰需要保持字面量精度。

```ts
const config = {
  theme: 'dark',
  size: 'md',
} as const
```

此时：

- `theme` 不再是 `string`
- 而是更精确的 `'dark'`
- 同时对象属性会被视为只读

常见注意点：`as const` 不是通用修正按钮。它适合常量配置、映射表、路由定义、静态枚举对象，不适合那些本来就需要修改的运行时对象。来源：[Literal Inference](https://www.typescriptlang.org/docs/handbook/2/everyday-types.html#literal-inference)

## 上下文类型推导

类型推导并不只来自右侧赋值，还可能来自“当前位置希望你写出什么类型”。

例如：

```ts
window.addEventListener('click', (event) => {
  console.log(event.clientX)
})
```

这里 `event` 的类型不是你手写的，而是由 `addEventListener` 的参数签名上下文推导出来的。

这叫上下文类型推导。它的价值很大：

- 回调函数参数很多时候都不用手动标注
- 复杂 API 调用中能自动给出更准确的提示
- 泛型 API 在良好设计下会自然形成强推导体验

来源：[Type Inference](https://www.typescriptlang.org/docs/handbook/type-inference.html)

## 函数返回值推导

TypeScript 也能根据函数体推导返回值类型。

```ts
function createUser() {
  return {
    id: '1',
    name: 'Ada',
  }
}
```

这对很多局部函数都非常方便。但在公共 API、工具库或复杂业务函数里，是否显式标注返回值就变成了一个设计问题。

更稳定的工程判断通常是：

- 局部、简单、显而易见的函数：可以依赖推导
- 公共函数、对外接口、复杂逻辑：显式标注返回值更稳

因为显式返回值不仅是给编译器看的，也是给协作者看的契约。

## 推导为什么有时会“太宽”

TypeScript 在推导时经常会偏向安全和可修改性，因此一些场景下会出现“你觉得应该更精确，但它推得更宽”的情况。

典型场景包括：

- 对象字面量属性
- 数组字面量
- `let` 变量
- 某些函数返回值未显式约束时

例如：

```ts
const list = ['a', 'b']
```

这通常是 `string[]`，而不是 `['a', 'b']`。如果你需要元组级精度，就必须显式提供更精确的信息，例如 `as const`。

## 最佳推导不是“全部交给编译器”

类型推导最容易走向两个极端：

- 极端一：什么都手写类型，代码冗长且重复
- 极端二：什么都不标，完全赌编译器能懂

更稳定的实践是：

- 局部简单值尽量依赖推导
- 对外接口显式标注
- 配置对象或静态映射用 `as const`
- 复杂函数返回值在需要时显式约束

类型推导是为了减少样板，不是为了放弃类型设计。

## 常见误区

### 误区一：编译器能推出来，就永远不用写类型

推导是方便，不是设计替代。公共函数和核心接口仍应明确声明返回值与输入边界。

### 误区二：对象字面量默认会保持最精确字面量类型

不准确。默认推导往往更宽，需要 `as const` 或显式类型约束。

### 误区三：as const 可以随便加

它会把结构变成更精确、更只读的形式。如果对象本来要修改，乱加 `as const` 只会带来后续冲突。

### 误区四：回调参数一定要手动标类型

很多时候上下文类型推导已经足够精确。过度手写只会增加重复代码。

## 推导在真实项目里的价值

理解类型推导后，你会更容易设计出好用的 API：

- 调用时尽量少写类型参数
- 返回值尽量保留精确结构
- 配置对象尽量保留字面量语义
- 让回调参数靠上下文自然获得精确类型

这也是 TypeScript 易用性和表达力的关键所在。

## 文末速查

| 场景           | 关键推导点                     |
| -------------- | ------------------------------ |
| `const` 基本值 | 更容易保留字面量类型           |
| `let` 变量     | 更容易宽化                     |
| 对象字面量     | 默认常常推得更宽               |
| 回调参数       | 常靠上下文类型推导             |
| 返回值         | 可推导，但公共接口常需显式标注 |
| 静态配置对象   | 常配合 `as const`              |

如果只保留一个结论，可以记住下面这句话：

> TypeScript 类型推导的关键，不是“能不能少写注解”，而是“什么时候让编译器自动推，什么时候由你显式收紧边界”。
