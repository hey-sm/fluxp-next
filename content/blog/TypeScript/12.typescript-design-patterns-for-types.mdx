# TypeScript 类型设计模式

学习 TypeScript 到一定阶段之后，真正的分水岭不再是“会不会写某个语法”，而是你是否具备类型设计能力。所谓类型设计，不是追求更复杂的泛型技巧，而是围绕真实业务边界，构造出稳定、可演化、可读的类型结构。

这篇文章讨论几种在前端项目和 TypeScript 工程中非常高频、也最有长期价值的类型设计模式。它们并不是语法特性本身，而是基于语言能力抽象出来的一套建模方法。

官方文档主线参考：

- [More on Objects](https://www.typescriptlang.org/docs/handbook/2/objects.html)
- [Unions and Intersections](https://www.typescriptlang.org/docs/handbook/unions-and-intersections.html)
- [Generics](https://www.typescriptlang.org/docs/handbook/2/generics.html)

## 为什么需要类型设计模式

很多团队会写 TypeScript，但类型层依然越来越乱，常见原因有三类：

- 一个类型到处复用，输入输出边界混在一起
- 所有对象都可变，导致共享数据被随意改写
- 状态和配置缺少结构化建模，分支越写越散

类型设计模式的价值，就是把这些高频问题沉淀成可复用的建模思路，而不是每次都从零开始凭直觉拼类型。

## 输入输出分离

一句话定义：输入输出分离是把“进入系统的数据类型”和“系统内部或输出给外部的数据类型”明确拆开，而不是用一个类型贯穿所有阶段。

它解决什么问题：很多数据在不同阶段语义并不相同。外部输入可能宽松、可空、命名不理想，而内部结构和输出结构往往需要更稳定、更规范。

什么时候用：

- 接口响应 DTO 与前端内部模型
- 表单原始值与提交 payload
- 组件 props 输入与事件回调输出

```ts
type UserDto = {
  id: string
  full_name: string
  avatar_url: string | null
}

type UserModel = {
  id: string
  name: string
  avatarUrl?: string
}
```

常见注意点：输入输出分离不是鼓励机械复制类型，而是在语义真正变化时保留转换边界。没有差异时不必强行拆层，但一旦存在语义差异，就不要继续共用一个类型。

## 只读边界

一句话定义：只读边界是在不该被修改的数据进入某个层级后，明确把它们限制为只读结构。

它解决什么问题：很多 bug 不是类型不对，而是某个本应被消费的数据被中途改写，导致共享状态和调用者预期失效。

什么时候用：

- 组件输入 props
- 配置对象
- store 导出的快照
- 领域模型读取层

```ts
type User = {
  id: string
  name: string
}

type UserView = Readonly<User>
```

常见注意点：只读边界的核心不是“所有东西都只读”，而是识别哪些层负责消费，哪些层负责修改。修改入口应该集中，消费入口应该尽量只读。

## 品牌类型

一句话定义：品牌类型是在结构相同的基础类型上附加一个只在类型层存在的标记，用于区分语义不同但底层表现相同的值。

它解决什么问题：很多业务值在运行时都是字符串或数字，但语义完全不同。例如 `UserId`、`OrderId`、`Token` 都可能是 `string`，如果全部混用，很难在编译期阻止误传。

什么时候用：

- 多种 id 都是字符串
- 金额、分数、百分比等都用数字表示
- 某些关键业务值需要额外区分语义身份

```ts
type Brand<T, Name extends string> = T & {
  readonly __brand: Name
}

type UserId = Brand<string, 'UserId'>
type OrderId = Brand<string, 'OrderId'>
```

常见注意点：品牌类型的价值在于建立语义区分，不在于复杂技巧。它适合关键边界值，不适合把所有基础类型都做品牌化，否则只会增加心智负担。

## 判别联合建模

一句话定义：判别联合建模是为每个状态分支设置稳定判别字段，再用联合类型表达完整状态空间。

它解决什么问题：如果业务状态靠多个可选字段拼装，最终很容易出现“不合法但类型上能通过”的中间状态。判别联合可以把状态机式逻辑建模得更精确。

什么时候用：

- 请求状态
- 异步流程
- 多模式组件
- 消息协议或事件系统

```ts
type LoadState =
  | { status: 'idle' }
  | { status: 'loading' }
  | { status: 'success'; data: string[] }
  | { status: 'error'; message: string }
```

常见注意点：判别联合的关键不是加一个 `status` 字段，而是确保每个状态分支只暴露该状态真正拥有的数据，避免“所有字段都可选”的宽松设计。

## 配置对象设计

一句话定义：配置对象设计是为组件、工具函数或模块的配置参数建立清晰的可选性、默认值语义和模式边界。

它解决什么问题：配置对象往往最容易出现“字段越来越多、语义越来越模糊、默认值与类型定义脱节”的问题。

什么时候用：

- 组件 props
- 工具函数 options
- SDK 初始化参数
- 表格、图表、表单等复杂配置项

```ts
type FetchOptions = {
  retry?: number
  timeout?: number
  mode?: 'json' | 'text'
}

function request(url: string, options: FetchOptions = {}) {
  const { retry = 0, timeout = 5000, mode = 'json' } = options
}
```

常见注意点：配置对象设计的重点不是字段越多越灵活，而是模式边界要清晰。对于互斥模式，通常应使用联合类型，而不是把所有字段都做成可选。

## 互斥模式建模

一句话定义：互斥模式建模用于表达“一个对象只能处于几种互斥形态之一”，常通过联合类型实现。

它解决什么问题：很多 API 并不是“任意字段组合”，而是“模式 A 用这组字段，模式 B 用另一组字段”。如果只用一个大对象加很多可选字段，调用方很容易传出非法组合。

```ts
type LinkButtonProps =
  | {
      kind: 'link'
      href: string
      onClick?: never
    }
  | {
      kind: 'action'
      onClick: () => void
      href?: never
    }
```

常见注意点：互斥模式设计的关键是让非法组合在编译期就无法成立，而不是在运行时靠文档提醒。

## 领域语义优先于技术写法

成熟的类型设计通常遵循一个原则：先考虑领域语义，再决定技术表达。也就是说：

- 先问数据在业务里代表什么
- 再问它在哪个阶段使用
- 再问是否需要可变、可选、可空、可复用
- 最后才决定用联合、泛型、工具类型还是品牌类型实现

如果顺序反过来，就很容易出现“语法很高级，但语义并不清晰”的类型设计。

## 类型设计的几个稳定原则

### 模型要服务数据流，而不是服务语法展示

一个类型如果脱离了数据进入、流转和输出的路径，就很难判断它是否合理。

### 输入边界可以宽，核心边界要稳

系统入口往往需要接纳不完整或未规范化的数据，但进入核心逻辑后，应尽快收紧为稳定结构。

### 修改入口要集中，消费入口要克制

这就是只读边界的重要性。不是所有层都应该既能读又能改。

### 允许模式分支时，要显式表达模式

不要让“互斥关系”藏在注释和约定里，应该用联合类型把它写进类型系统。

## 常见误区

### 误区一：一个类型能复用越多地方越好

复用过度往往意味着语义被稀释。真正稳定的是“边界清晰的适度复用”，而不是一个类型包打天下。

### 误区二：品牌类型是高级技巧，项目里最好都用上

品牌类型适合关键语义区分，不适合作为默认建模方式。它是精确工具，不是普适模板。

### 误区三：配置对象全做可选最灵活

这会把很多非法组合交给运行时处理。真正成熟的配置设计，应该能在类型层表达默认值语义和互斥模式。

### 误区四：类型设计只影响编译器，不影响架构

不准确。类型边界本身就是架构边界的可执行表达，尤其在组件 API、状态管理和领域模型中更明显。

## 文末速查

| 问题                            | 优先想到的设计模式   |
| ------------------------------- | -------------------- |
| 外部输入和内部模型语义不同      | 输入输出分离         |
| 某层不该随意修改数据            | 只读边界             |
| 不同字符串 / 数字值语义需要区分 | 品牌类型             |
| 一个对象存在明确状态分支        | 判别联合             |
| 配置项存在默认值与可选性问题    | 配置对象设计         |
| 不同模式字段互斥                | 联合类型建模互斥模式 |

如果只保留一个结论，可以记住下面这句话：

> TypeScript 真正成熟的使用方式，不是把类型写得更复杂，而是让类型结构与业务边界、数据流阶段和修改权限保持一致。
