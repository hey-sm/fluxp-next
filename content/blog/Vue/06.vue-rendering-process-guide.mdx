# Vue 3 渲染流程详解

要真正理解 Vue 组件是如何工作的，不能只停留在“数据变了，页面就更新了”这句口号上。更重要的是弄清它中间到底经历了哪些阶段：模板如何变成渲染逻辑，响应式状态变化后为什么不会立刻同步改 DOM，Vue 又是如何找到最小必要更新范围的。

Vue 官方在渲染机制文档里，把这条链路概括得很清晰：组件挂载时会经历编译、挂载、更新三步；更新本身又建立在响应式副作用重新运行与运行时 patch 之上。理解这条链路后，很多开发中的现象都会变得可解释。

来源：[Rendering Mechanism](https://vuejs.org/guide/extras/rendering-mechanism)

## Vue 渲染流程的整体结构

从高层看，一次 Vue 组件的首次渲染大致是：

```text
模板
  -> 编译成渲染函数
  -> 执行渲染函数生成 VNode
  -> 挂载真实 DOM
```

后续更新则是：

```text
响应式依赖变化
  -> 组件渲染 effect 重新执行
  -> 生成新的 VNode
  -> 新旧 VNode 比较
  -> patch 到真实 DOM
```

也就是说，Vue 的页面更新从来不是“直接改模板”，而是围绕渲染函数、虚拟节点和运行时 patch 展开。

## 模板编译：为什么 Vue 能写模板

Vue 模板不是浏览器原生能力，而是编译器提供的上层语法。官方文档明确说明，模板会被编译成返回虚拟 DOM 树的渲染函数。

这一步的意义在于：

- 开发时写更接近 HTML 的模板
- 运行时真正执行的是更底层的渲染逻辑
- 编译器可以在构建阶段注入静态优化信息

因此，模板并不是和渲染函数平行的两套运行机制，而是更适合人类编写的渲染函数前端语法。来源：[Rendering Mechanism](https://vuejs.org/guide/extras/rendering-mechanism)

## VNode：Vue 更新的中间表示

Vue 的运行时不会直接拿模板去改 DOM，而是先生成虚拟节点，也就是 `VNode`。可以把它理解成“当前 UI 结构的内存表示”。

它至少承载这些信息：

- 节点类型
- props / attributes
- children
- 某些编译期优化标记

VNode 的作用，不是替代 DOM，而是给渲染器提供一层可比较、可 patch 的中间模型。

## 首次挂载发生了什么

组件首次挂载时，运行时渲染器会：

1. 执行渲染函数
2. 得到当前组件的 VNode 树
3. 遍历这棵树
4. 创建真实 DOM 节点
5. 将其插入页面

Vue 官方强调，这一步会作为响应式副作用执行，因此其中读取到的响应式状态会被自动追踪。这意味着组件的 render 本身就是一个 effect。

这也是为什么模板里读取了哪些响应式值，会直接决定后续哪些变化能触发这个组件更新。

## 组件更新为什么会发生

一旦组件渲染过程中读取过的某个响应式依赖变化，Vue 就会重新调度该组件的渲染副作用。于是：

- 组件 render 再次执行
- 生成新的 VNode 树
- 渲染器把新旧 VNode 做比较
- 只把必要的变化 patch 到 DOM

这说明一个很重要的事实：

- Vue 更新的直接原因不是“模板里写了表达式”
- 而是“当前组件的渲染副作用订阅了哪些响应式依赖”

## patch：Vue 是怎么更新 DOM 的

当新的 VNode 树生成后，Vue 不会销毁整棵旧 DOM 再重建，而是通过 patch 过程比较新旧节点差异，只应用必要的变更。

这个 patch 过程通常关注：

- 节点类型是否变化
- props 是否变化
- children 是否变化
- 列表节点如何复用

因此，Vue 的高效更新依赖的不是“神奇地不重新渲染”，而是：

- 组件更新范围尽量精确
- VNode 比较尽量高效
- DOM patch 尽量最小化

## 编译器为什么对性能重要

Vue 和很多纯运行时框架的一个关键差异，在于它的模板编译器会注入编译时优化信息。官方文档明确提到三类很重要的优化：

- 缓存静态内容
- patch flags
- 树结构打平（tree flattening）

### 缓存静态内容

如果一段模板完全不依赖动态绑定，编译器可以把这部分静态节点缓存起来，后续更新时直接复用，而不必每次重新创建和比较。

### patch flags

编译器会在 vnode 创建调用中编码“这个节点未来可能更新什么”。这样运行时在 patch 时，不必对所有维度都做全量检查，而可以更聚焦地比较真正可能变化的部分。

### 树结构打平

编译器会让更新路径更聚焦于动态节点集合，减少无意义遍历。

这也是为什么 Vue 模板不只是“写起来方便”，而且天然带有编译期优化红利。来源：[Rendering Mechanism](https://vuejs.org/guide/extras/rendering-mechanism)

## nextTick 为什么存在

很多人刚接触 Vue 时，会误以为状态一改，DOM 就已经立刻是新的。实际上，Vue 会把同一轮里的多个状态修改合并起来统一刷新。

这带来一个直接结果：

- 你在改完状态后立刻读取 DOM，拿到的可能还是旧 DOM

这时就需要 `nextTick()`：

```vue
<script setup lang="ts">
import { nextTick, ref } from 'vue'

const open = ref(false)

async function expand() {
  open.value = true
  await nextTick()
  console.log('此时可以读取更新后的 DOM')
}
</script>
```

`nextTick` 的作用不是“强制刷新”，而是等待当前这轮 DOM 更新完成后再继续。来源：[Reactivity Fundamentals](https://vuejs.org/guide/essentials/reactivity-fundamentals)

## 批处理：为什么多次改状态不会多次立刻刷 DOM

Vue 会把同步代码块里的多次状态变化合并调度，这叫更新批处理。它的目的非常明确：

- 避免每次状态变化都立刻触发一次完整渲染
- 把多次修改压缩到一轮更高效的更新中

例如：

```ts
count.value++
count.value++
count.value++
```

从业务角度看，用户最终只需要看到一次正确结果，不需要为中间每个值都单独做一轮 DOM 更新。这种批处理机制正是现代前端框架保证性能的基本能力之一。

## 为什么有时“组件更新了”，但你感觉没变化

组件 render 重新执行并不一定意味着 DOM 发生了明显变化。原因通常有几类：

- 生成的新 VNode 和旧结果在关键部分一致
- 某些静态节点被缓存复用
- patch flag 告诉运行时无需全量对比
- 依赖变化了，但最终模板输出没变

这也是为什么调试性能时，不能简单把“组件函数重新运行”直接等价成“页面卡顿”。真正昂贵的通常是：

- 大量不必要的组件更新
- 大量复杂子树 patch
- 列表更新策略不稳定

## 模板、渲染函数和 JSX 的关系

官方文档明确说明，模板最终会编译为渲染函数。Vue 同时也提供手写渲染函数与 JSX 的能力，但默认仍然推荐模板。

原因很直接：

- 模板更接近 HTML
- 更适合静态分析
- 能更充分地利用编译优化

渲染函数和 JSX 更适合：

- 极高度动态的可复用组件
- 需要完全用 JavaScript 组织视图结构的场景

但它们不是大多数业务组件的默认主线。来源：[Render Functions & JSX](https://vuejs.org/guide/extras/render-function)

## 工程上怎么理解这条渲染链路

如果把 Vue 渲染流程压缩成工程判断，可以记住下面几条：

- 模板不是最终运行形式，渲染函数才是
- render 读取响应式值时会建立依赖关系
- 状态变化后不会立刻同步刷 DOM，而是进入更新调度
- 更新的核心是新旧 VNode 比较和 patch
- 编译器优化直接决定运行时 patch 是否足够精确

理解这套链路后，以下问题会更容易判断：

- 为什么要用 `nextTick`
- 为什么列表 `key` 很重要
- 为什么静态节点复用对性能有价值
- 为什么某些组件虽然 render 了，但 DOM 成本并不高

## 常见误区

### 改状态后 DOM 一定立刻更新

不准确。Vue 会做批处理和异步刷新调度。

### 模板就是运行时直接解释执行

不准确。模板通常会先编译成渲染函数。

### 一次组件更新就代表整个子树都重建

不准确。Vue 的运行时 patch 会尽量只更新必要部分。

### 性能优化只看模板写法

不完整。真正影响性能的是响应式依赖边界、组件结构、列表更新策略和编译期优化结果。

## 文末速查

| 阶段       | 含义                                  |
| ---------- | ------------------------------------- |
| 编译       | 模板转成渲染函数                      |
| 首次挂载   | 执行 render，生成 VNode，创建真实 DOM |
| 更新       | 响应式依赖变化后重新执行 render       |
| patch      | 新旧 VNode 比较并最小化更新 DOM       |
| `nextTick` | 等待当前这轮 DOM 更新结束后再继续     |

如果只保留一个结论，可以记住下面这句话：

> Vue 的渲染流程本质上是“模板编译 + 响应式驱动的 render effect + VNode patch”，而不是状态变化后直接命令式改 DOM。
