# Vue 3 性能优化详解

性能优化在 Vue 项目里最常见的误区，不是“不会用某个技巧”，而是太早陷入零散技巧。真正有效的优化，首先要分清问题发生在哪一层：

- 页面加载阶段
- 组件更新阶段
- 大列表 / 大子树渲染阶段
- 应用结构与资源拆分阶段

Vue 官方性能文档也基本沿着这条线展开。换句话说，性能优化不是一堆 API 小技巧，而是一套判断框架。

来源：[Performance](https://vuejs.org/guide/best-practices/performance)

## 先分清两类性能问题

### 页面加载性能

关注的是：

- 首屏资源体积
- 首次内容显示时间
- 页面代码是否被过早打包进首屏

### 更新性能

关注的是：

- 某次状态变化会影响多少组件
- 哪些子树是无效更新
- 列表和复杂区域是否被反复重算与重 patch

不把这两类问题分开，就很容易出现“明明是首屏包过大，却在组件里狂加 `computed`”这类无效优化。

## 页面加载阶段：先看资源结构

对客户端应用而言，很多时候最大问题不是 patch 速度，而是首屏下载了太多本不需要立即执行的代码。

更稳定的优化主线通常是：

- 路由懒加载
- 异步组件
- 减少不必要的大型依赖

### 路由懒加载

页面级组件应优先使用动态导入拆包：

```ts
const routes = [
  {
    path: '/reports',
    component: () => import('@/pages/ReportsPage.vue'),
  },
]
```

这通常比在首屏包里一次性装下所有页面更有效。来源：[Lazy Loading Routes](https://router.vuejs.org/guide/advanced/lazy-loading)

### 异步组件

当某个组件不是首屏必须内容时，可以用异步组件延迟加载。

```ts
import { defineAsyncComponent } from 'vue'

const HeavyChart = defineAsyncComponent(() => import('./HeavyChart.vue'))
```

这类优化尤其适合：

- 图表
- 编辑器
- 预览器
- 设置面板

## 更新性能：先看更新范围，而不是先上技巧

Vue 官方性能文档里有一个非常重要的原则：**props 稳定性**。

如果父组件每次更新时都给子组件传入新的、不稳定的输入，那么子组件就更容易重新更新。与其在子组件里做复杂优化，不如先让输入边界稳定下来。

例如，把“当前项是否激活”的判断提前在父级算好，再把结果作为稳定布尔值传给子项，通常比让每个子项内部自己判断更高效。来源：[Performance](https://vuejs.org/guide/best-practices/performance)

## 计算属性缓存的价值

`computed` 的价值不只是“写起来优雅”，更在于它对派生值提供缓存语义。Vue 官方还特别提到，在 Vue 3.4+ 中，如果 `computed` 返回值没有变化，就不会继续触发依赖它的 effect。

这意味着：

- 适合写可重复读取的派生结果
- 不应把本该是 `computed` 的逻辑写进模板表达式里反复执行

```ts
const activeUserIds = computed(() =>
  users.value.filter((user) => user.active).map((user) => user.id),
)
```

如果某个派生值会被模板或多个逻辑反复使用，把它提到 `computed` 通常是最自然的优化。来源：[Performance](https://vuejs.org/guide/best-practices/performance)

## v-once

一句话定义：`v-once` 用来告诉 Vue：这段子树只渲染一次，后续更新都跳过。

它解决什么问题：当某块内容依赖运行时值完成第一次渲染后就再也不会变时，没必要让它继续参与后续更新。

```vue
<template>
  <section v-once>
    <h2>{{ appName }}</h2>
  </section>
</template>
```

常见注意点：`v-once` 适合“真的不会再变”的内容，不适合“只是当前看起来变化不多”的内容。否则会直接冻结后续更新。来源：[Performance](https://vuejs.org/guide/best-practices/performance)

## v-memo

一句话定义：`v-memo` 用来在满足记忆条件时跳过大子树或列表片段的更新。

它解决什么问题：某些大块模板结构只有在少数关键依赖变化时才需要更新，此时 `v-memo` 可以减少无意义 patch。

```vue
<template>
  <div v-for="item in items" :key="item.id" v-memo="[item.id === activeId]">
    {{ item.name }}
  </div>
</template>
```

常见注意点：`v-memo` 是针对大型子树或大列表的有针对性优化，不应无差别铺满模板。它的前提是你已经明确知道哪些依赖决定这块子树是否需要更新。来源：[Performance](https://vuejs.org/guide/best-practices/performance)

## 大列表优化

大列表的性能问题通常不只是“渲染多”，而是：

- 渲染项数量太多
- 每一项输入不稳定
- 每一项内部逻辑过重
- 滚动时重复创建和 patch 成本高

稳定的优化顺序通常是：

1. 保证 `key` 稳定
2. 保证列表项 props 稳定
3. 减少列表项内部复杂计算
4. 必要时做列表虚拟化

Vue 官方性能文档明确提到，对大列表应考虑 virtualization，只渲染当前视口附近的部分项，而不是全量挂到 DOM。来源：[Performance](https://vuejs.org/guide/best-practices/performance)

## 组件抽象并不是越多越好

很多项目的性能问题并不来自 Vue 不够快，而来自组件切分粒度失控：

- 一个很小的结构被拆成很多极薄组件
- 父层状态变一下，大量子组件都要参与更新
- 组件层级过深增加了不必要的更新传播范围

组件抽象的目标应该是边界清晰，而不是“越细越先进”。过度抽象既影响性能，也影响可读性。

## 条件渲染的成本判断

和前面模板文章呼应一下：

- `v-if` 适合切换不频繁、初始不必渲染的区域
- `v-show` 适合频繁显隐的区域

这本质上也是性能判断问题。选择错误，会导致要么初始渲染过重，要么频繁创建销毁子树。

## 什么时候该优先优化数据流，而不是优化模板

如果一个页面开始变慢，通常先查这几件事：

- 是不是父层持有了过大的响应式对象
- 是不是大量状态本可局部化却被提升过高
- 是不是派生逻辑写在模板里反复执行
- 是不是列表项接收了不稳定对象 / 函数输入

这些问题本质上都不是“模板技巧不足”，而是数据流边界和更新范围设计不合理。

## 常见误区

### 一看到卡顿就先加 v-memo

这往往跳过了真正的问题定位。应先判断是加载、更新、列表还是资源结构问题。

### 所有派生逻辑都直接写模板里

短期可行，但会让复杂页面中同一逻辑被频繁重复求值。

### 页面组件不做懒加载

客户端应用越大，首屏越容易被无关页面拖慢。

### 列表项 key 不稳定

这会让 patch 成本和状态稳定性同时变差。

### 把性能优化等同于“少写组件”

真正重要的是组件边界是否稳定，而不是数量本身。

## 文末速查

| 问题类型                   | 优先排查方向                         |
| -------------------------- | ------------------------------------ |
| 首屏慢                     | 路由懒加载、异步组件、依赖体积       |
| 更新慢                     | props 稳定性、状态边界、派生逻辑位置 |
| 大列表卡                   | key、列表项复杂度、虚拟化            |
| 永不变化的运行时内容       | `v-once`                             |
| 大子树或大列表局部跳过更新 | `v-memo`                             |

如果只保留一个结论，可以记住下面这句话：

> Vue 性能优化的核心不是堆技巧，而是先分清问题发生在“资源加载”还是“更新传播”，再针对性缩小成本范围。
