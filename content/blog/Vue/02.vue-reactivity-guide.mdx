# Vue 3 响应式系统详解

Vue 之所以能做到“数据变了，界面跟着变”，核心并不在模板语法本身，而在它背后的响应式系统。模板、组件、计算属性、监听器、组件更新，最终都建立在同一套响应式依赖追踪模型上。

如果没有对这套模型的清晰认知，Vue 代码很容易停留在“能跑”的层面，而一旦遇到这些问题就会开始混乱：

- 为什么有的值改了页面会更新，有的不会
- 为什么解构后数据不响应了
- 为什么 `watch` 能监听到变化，`computed` 却不适合做副作用
- 为什么大对象、深层嵌套和大列表有时会带来性能开销

这篇文章的重点不是源码逐行分析，而是建立一套足够稳定的工程心智模型。

## 响应式系统到底在做什么

可以把 Vue 响应式系统理解为一套自动依赖管理机制。它至少完成三件事：

1. 追踪“谁读取了哪个响应式值”
2. 在值变化时找到相关依赖
3. 重新执行依赖这些值的副作用或渲染逻辑

在 Vue 组件里，最典型的依赖方包括：

- 组件渲染函数
- `computed`
- `watchEffect`
- `watch` 内部依赖追踪部分

也就是说，Vue 不是在“侦测一切变化后全量刷新页面”，而是在维护一张依赖关系图。

## Vue 3 的基本响应式链路

从概念上看，一次典型更新大致经历这条链路：

```text
响应式值被读取
  -> Vue 记录依赖关系
响应式值被修改
  -> Vue 找到相关依赖
  -> 调度这些依赖重新运行
  -> 组件或副作用得到更新
```

放到组件渲染里，可以进一步理解成：

```text
模板渲染时读取了 count
  -> 组件渲染 effect 订阅 count
count 变化
  -> 渲染 effect 被重新调度
  -> 组件生成新的 VNode
  -> patch 更新真实 DOM
```

这就是“声明式 UI”的底层基础：你不手动写 DOM 更新步骤，而是让依赖关系驱动渲染系统完成更新。

## ref 和 reactive 是两种不同的响应式入口

虽然 `ref` 和 `reactive` 都属于响应式 API，但它们表达的是两类不同状态模型。

### ref：面向单值容器

`ref` 适合表达“一个单独的响应式槽位”。它可以持有原始类型，也可以持有对象。

```ts
const count = ref(0)
const user = ref({ id: 1, name: 'Ada' })
```

它的优势在于：

- 语义直接
- 替换整个值时边界清晰
- 在组合式函数里更容易作为明确返回值暴露

### reactive：面向对象代理

`reactive` 更适合“一整块对象状态”。

```ts
const form = reactive({
  name: '',
  email: '',
  agreed: false,
})
```

它的优势在于：

- 对对象字段的操作更自然
- 表达整块状态更紧凑
- 在模板里读写更接近普通对象

### 什么时候该选谁

可以用一个简单判断：

- 如果你想强调“这是一个单独值”或“这个值经常整体替换”，优先 `ref`
- 如果你想表达“这是一整块关联字段”，优先 `reactive`

在真实项目里，最稳定的实践通常不是“坚持只用一种”，而是根据状态边界混合使用。

## 依赖是如何被收集的

Vue 3 的响应式依赖收集是运行时自动完成的。核心前提是：

- 当某个响应式值在活跃 effect 中被读取时，Vue 会记录这条依赖关系

例如：

```ts
const count = ref(0)

const doubled = computed(() => count.value * 2)
```

当 `computed` 的 getter 运行时，读取了 `count.value`。于是 Vue 会记录：

- `doubled` 依赖 `count`

之后只要 `count` 变化，`doubled` 就会被标记为需要重新计算。

这套机制的关键不是“手动声明依赖数组”，而是**读取即订阅**。这也是 Vue 和很多需要显式依赖声明的系统在使用体验上的核心差异之一。来源：[Reactivity in Depth](https://vuejs.org/guide/extras/reactivity-in-depth.html)

## 修改值后为什么不会立刻同步刷新 DOM

很多人刚接触 Vue 时，会误以为“改状态 = 立刻改 DOM”。实际上，Vue 会把同一轮事件循环里的多次更新合并调度，再批量应用到渲染流程中。

这意味着：

- 多次同步改状态不会对应多次立即 DOM 操作
- Vue 会做更新批处理
- 如果你在更新后立刻读取 DOM，可能拿到的还是旧 DOM，需要等待下一次刷新周期

这也是 `nextTick()` 存在的原因之一。它不是“强制更新”，而是“等待当前这轮 DOM 更新完成后再继续做事”。来源：[Reactivity Fundamentals](https://vuejs.org/guide/essentials/reactivity-fundamentals)

## 为什么 computed、watch、组件更新都建立在同一套响应式基础上

从工程视角看，它们只是“不同用途的 effect”。

### computed

`computed` 是带缓存的派生 effect。它最适合纯计算，不适合副作用。

### watch / watchEffect

`watch` 和 `watchEffect` 是副作用 effect。它们的目标不是返回值，而是响应状态变化去执行额外逻辑。

### 组件渲染

组件渲染本身也是一个 effect。模板里读取到的响应式值，会成为该组件更新的依赖。

把这三者放到一起看，很多判断就会自然清晰：

- “派生结果”放 `computed`
- “状态变化后要做某事”放 `watch`
- “界面展示”交给组件 render

## 深响应与浅响应

Vue 默认的对象响应式通常是深层的。也就是说，如果你把一个嵌套对象交给 `reactive`，内部嵌套字段通常也会被纳入响应式系统。

这在大多数业务场景里是方便的，但也要理解它的代价：

- 数据结构越深，代理和依赖追踪范围越大
- 大型不可变数据结构如果整个深层都响应式化，可能带来额外开销

因此 Vue 也提供了浅层版本，例如：

- `shallowRef`
- `shallowReactive`

它们的意义不是“更高级”，而是缩小响应式边界，只追踪最外层引用变化。

```ts
import { shallowRef } from 'vue'

const rows = shallowRef([{ id: 1, name: 'Ada' }])

rows.value = [...rows.value, { id: 2, name: 'Linus' }]
```

这类做法更适合：

- 大型不可变数据列表
- 外部状态系统或第三方数据结构接入
- 你只关心整体替换而不关心内部逐字段追踪

来源：[Performance](https://vuejs.org/guide/best-practices/performance)

## 常见误区一：解构 reactive 后丢失响应式连接

这是业务开发里最常见的问题之一。

```ts
const state = reactive({
  count: 0,
})

const { count } = state
```

这里解构出来的 `count` 不再保留原本的响应式访问语义。它只是当前值的普通拷贝，不会继续跟着 `state.count` 自动同步。

更稳定的做法通常是：

- 直接用 `state.count`
- 或者在需要解构暴露时使用 `toRefs`

## 常见误区二：把可计算逻辑写成 watch

例如：

```ts
const firstName = ref('Ada')
const lastName = ref('Lovelace')
const fullName = ref('')

watch([firstName, lastName], () => {
  fullName.value = `${firstName.value} ${lastName.value}`
})
```

这段代码技术上可以工作，但语义是错位的。`fullName` 明明是派生结果，却被写成了副作用同步。

更合适的写法应该是：

```ts
const fullName = computed(() => `${firstName.value} ${lastName.value}`)
```

一条很重要的判断原则是：

- 可直接推导出的值，用 `computed`
- 需要触发副作用的变化响应，用 `watch`

## 常见误区三：用 reactive 持有经常整体替换的数据

如果你的状态天然是“一次次整体替换”，例如列表查询结果、表格数据页、接口返回块，那么用 `ref` 持有整个对象或数组通常比 `reactive` 更稳。

```ts
const rows = ref<User[]>([])

async function fetchRows() {
  rows.value = await requestUsers()
}
```

这种写法的边界更清楚，也更接近“请求返回一份新数据”的真实语义。

## 常见误区四：把一切都做成深响应

深响应并不是越多越好。对以下场景，应优先考虑缩小响应式范围：

- 大型静态配置对象
- 只会整体替换的后端返回列表
- 第三方类实例
- 明显不需要逐字段追踪的只读结构

响应式系统是手段，不是目标。它越精准，系统越稳定。

## 工程中的推荐组织方式

如果从项目实践角度给出一个稳定建议，可以按下面方式组织：

- 组件内局部简单状态：优先 `ref`
- 一组强关联表单或筛选条件：优先 `reactive`
- 派生值：`computed`
- 与外部系统同步、异步请求、日志：`watch`
- 依赖项很多且副作用轻量：`watchEffect`
- 大型不可变数据或外部对象：优先评估 `shallowRef` / `shallowReactive`

这套边界一旦稳定，后续调试和性能判断都会清晰很多。

## 文末速查

| 问题                             | 判断方式                                       |
| -------------------------------- | ---------------------------------------------- |
| 这个值变了为什么不更新           | 先确认它是否是响应式来源                       |
| 这个逻辑该写 computed 还是 watch | 能推导就 `computed`，有副作用才 `watch`        |
| 为什么解构后不响应了             | `reactive` 解构会丢失原始代理访问              |
| 为什么更新后 DOM 没立刻变        | Vue 对更新做了批处理，需要等刷新周期           |
| 大列表或大对象为什么有时性能一般 | 响应式边界过大，需考虑浅响应或更稳定的数据结构 |

如果只保留一个结论，可以记住下面这句话：

> Vue 响应式系统的核心不是“自动刷新页面”，而是“自动维护状态与依赖之间的关系图”。
