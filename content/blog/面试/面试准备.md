# 浏览器

## Js单线程

> JavaScript 之所以是单线程的，是因为 JavaScript 的设计初衷是为了处理网页中的交互逻辑，而网页中的交互通常是基于事件驱动的。因此，为了避免复杂的线程同步和数据竞争问题，JavaScript 的设计者选择了单线程模型。
>
> 虽然 JavaScript 是单线程的，但浏览器中还有其他线程来处理不同的任务，例如渲染线程用于页面渲染、网络线程用于处理网络请求、定时器线程用于处理定时器事件等。这些线程与 JavaScript 主线程是分离的，它们之间通过事件循环机制进行通信和协作。
>
> 单线程模型意味着 JavaScript 代码中的任务是按顺序执行的，任何一个任务的执行都会阻塞后续任务的执行。为了避免长时间的任务阻塞页面的渲染和响应，JavaScript 中通常会使用异步编程模式，例如回调函数、Promise、async/await 等，来将耗时的任务转化为异步执行，从而保持页面的流畅性和响应性。

## 浏览器渲染流程

> 浏览器的渲染流程通常包括以下几个步骤：
>
> 1. **构建 DOM 树：** 浏览器从 HTML 文档中解析出 DOM 树（文档对象模型），这是页面的结构表示。浏览器会逐行解析 HTML，将每个 HTML 元素转换为 DOM 节点，并根据它们的嵌套关系构建树形结构。
> 2. **构建 CSSOM 树：** 浏览器解析 CSS 样式表，将样式规则转换为 CSSOM 树（CSS 对象模型）。类似于 DOM 树，CSSOM 树表示了样式的层级结构，其中包含了每个元素的样式信息。
> 3. **合并 DOM 树和 CSSOM 树：** 浏览器将 DOM 树和 CSSOM 树合并成一个结合体，这个结合体被称为渲染树（Render Tree）。渲染树中包含了所有需要呈现在页面上的可见元素以及它们的样式信息。
> 4. **布局：** 浏览器根据渲染树中每个节点的大小、位置和样式信息，计算出每个节点在页面上的准确位置，这个过程被称为布局或回流（reflow）。浏览器会确定每个节点的大小和位置，并计算它们之间的相互关系。
> 5. **绘制：** 浏览器使用计算出的布局信息，将渲染树中的每个节点转换为屏幕上的实际像素，这个过程被称为绘制（painting）。浏览器会遍历渲染树中的每个节点，并将它们转换为屏幕上的像素，然后绘制到屏幕上。

## 事件循环机制

浏览器事件循环（Event Loop）是浏览器处理 JavaScript 代码的机制，它负责管理 JavaScript 代码的执行顺序、异步任务的执行以及事件的处理等。以下是浏览器事件循环机制的基本流程：

1. **执行栈（Execution Stack）：** 所有的 JavaScript 代码都运行在执行栈中。当浏览器执行 JavaScript 代码时，会将代码按照顺序压入执行栈中，并依次执行。执行栈遵循“先进后出”的原则，即最后压入执行栈的代码最先执行。

2. **异步任务的注册和执行：** 当 JavaScript 遇到异步任务（例如定时器、事件监听器、网络请求等）时，会将任务注册到相应的事件队列中，而不是立即执行。当异步任务的条件满足时（例如定时器时间到、事件触发等），浏览器会将任务添加到任务队列中。

3. **任务队列（Task Queue）：** 浏览器有多个任务队列，例如宏任务队列（Macro Task Queue）和微任务队列（Micro Task Queue）。宏任务队列包括 setTimeout、setInterval、事件监听器等，而微任务队列包括 Promise、MutationObserver 等。在每一轮事件循环中，浏览器会首先处理微任务队列中的任务，然后处理宏任务队列中的任务。

4. **事件循环（Event Loop）：** 浏览器会不断地重复执行以下步骤，直到执行栈为空：
   - 从宏任务队列中取出一个任务，压入执行栈中执行。
   - 执行栈中的代码执行完毕后，检查微任务队列中是否有任务，如果有则依次执行微任务队列中的任务。
   - 如果微任务队列中的任务执行完毕，浏览器会渲染页面，然后开始下一轮事件循环。

5. **渲染和交互：** 在事件循环的过程中，浏览器会根据 JavaScript 代码的执行情况和页面的变化来进行渲染。如果需要更新页面，则浏览器会将页面的变化显示到用户界面上，并处理用户的交互事件。

总的来说，浏览器事件循环机制负责管理 JavaScript 代码的执行顺序、异步任务的执行以及页面的渲染和交互。了解事件循环机制有助于开发者更好地理解 JavaScript 代码的执行过程，并优化代码的性能和用户体验。

## 数据存储方案

前端数据存储是指在浏览器端存储数据的一种机制，用于在不同页面间共享数据、实现持久化存储以及提高用户体验。常见的前端数据存储方式包括：

> 1. **Cookies（Cookie）：** Cookies 是最古老和最广泛使用的前端数据存储方式之一。它可以存储少量的数据，并且可以在客户端和服务器端之间进行传递。Cookies 的缺点是存储容量有限（一般为几 KB），且需要手动设置过期时间。
> 2. **Web Storage（Web 存储）：** Web 存储包括 localStorage 和 sessionStorage 两种机制。localStorage 用于持久化存储数据，数据保存在浏览器关闭后仍然存在；而 sessionStorage 的数据在浏览器关闭后会被清除。Web 存储可以存储大量数据（一般为几 MB），并且可以通过 JavaScript API 进行操作。
> 3. **IndexedDB：** IndexedDB 是一种低级别的浏览器数据库，提供了一个异步的 API 来存储和检索结构化数据。IndexedDB 允许存储大量数据，并且支持事务操作、索引、游标等功能，适用于复杂的数据存储需求。
> 4. **Cache Storage：** Cache Storage 是 Service Worker 的一个功能，用于存储缓存的响应数据，以便在离线时提供给用户。Cache Storage 可以存储网络请求的响应数据、静态资源文件等，并且支持自定义缓存策略。
> 5. **Session Storage：** Session Storage 与 Web Storage 的 sessionStorage 相似，但它是与会话相关联的。数据存储在浏览器会话期间，如果用户关闭了标签页或者浏览器窗口，数据会被清除。

**Cookies、Web Storage、IndexedDB 和 Cache Storage 的对比：**

- Cookies 可以存储少量的数据，但是每次请求都会携带在 HTTP 请求头中，影响性能；而 Web Storage、IndexedDB 和 Cache Storage 可以存储大量的数据，并且不会在每次请求中携带。
- Web Storage 和 IndexedDB 只能存储字符串类型的数据，而 Cache Storage 可以存储 Response 对象和其他类型的数据。
- IndexedDB 提供了更强大的查询和事务支持，适用于复杂的数据存储需求，而 Cache Storage 主要用于缓存网络请求的响应数据。

## 防抖节流

防抖（Debounce）和节流（Throttle）是用于控制函数执行频率的两种常见技术。它们的主要区别在于对函数执行的时间间隔的控制方式不同。

1. **防抖（Debounce）：** 在一段连续的操作中，只有最后一次操作的间隔超过指定时间后，才会执行函数。如果在指定时间内又有操作发生，则重新开始计时。适用于如搜索框输入联想、滚动事件等场景。

2. **节流（Throttle）：** 在一段连续的操作中，以固定的时间间隔执行函数。即使在执行函数期间有多次操作发生，也只会在每个固定的时间间隔内执行一次函数。适用于如页面滚动、鼠标移动等持续触发的事件。

下面是防抖和节流的简单实现示例：

```javascript
// 防抖函数
function debounce(func, delay) {
  let timer
  return function (...args) {
    clearTimeout(timer)
    timer = setTimeout(() => {
      func.apply(this, args)
    }, delay)
  }
}

// 节流函数
function throttle(func, delay) {
  let canRun = true
  return function (...args) {
    if (!canRun) return
    canRun = false
    setTimeout(() => {
      func.apply(this, args)
      canRun = true
    }, delay)
  }
}
```

使用示例：

```javascript
// 防抖
const debouncedFunc = debounce(() => {
  console.log('Search triggered')
}, 300)

// 节流
const throttledFunc = throttle(() => {
  console.log('Scrolling')
}, 200)

// 添加事件监听
window.addEventListener('scroll', throttledFunc)
```

在上面的示例中，`debounce` 函数用于创建一个防抖函数，`throttle` 函数用于创建一个节流函数。然后，我们可以将需要进行防抖或节流处理的函数传入这些函数中，以便在事件触发时进行调用。

## Promise

Promise 是 JavaScript 中用于处理异步操作的对象，它表示一个异步操作的最终完成或失败，并且可以通过链式调用来处理异步操作的结果。

> 一个 Promise 可以处于以下三种状态之一：
>
> - 进行中（Pending）：Promise 对象刚被创建，还未完成也未失败。
> - 已完成（Fulfilled）：异步操作成功完成。
> - 已失败（Rejected）：异步操作失败。
>
> Promise 对象具有以下特点：
>
> 1. **异步操作：** Promise 主要用于处理异步操作，比如网络请求、文件读取等。
> 2. **链式调用：** 可以通过 `.then()` 方法链式调用多个处理函数，依次处理异步操作的结果。
> 3. **状态不可逆：** 一旦 Promise 对象的状态发生改变，就会保持不变，无法再次改变。

## Proxy

Proxy 是 JavaScript 中的一个内置对象，它允许你创建一个代理对象，用于修改目标对象的默认行为。代理对象可以拦截并重定义底层操作，比如属性查找、赋值、函数调用等。

Proxy 的基本语法如下：

```javascript
const proxy = new Proxy(target, handler)
```

- `target`：要代理的目标对象。
- `handler`：一个包含各种代理行为的对象，用于定义代理对象的行为。

下面是一个简单的示例，演示了如何使用 Proxy 来拦截对象的属性访问和赋值操作：

```javascript
const target = {
  name: 'John',
  age: 30,
}

const handler = {
  get: function (target, prop) {
    console.log(`Getting property "${prop}"`)
    return target[prop]
  },
  set: function (target, prop, value) {
    console.log(`Setting property "${prop}" to "${value}"`)
    target[prop] = value
  },
}

const proxy = new Proxy(target, handler)

// 通过代理对象访问属性
console.log(proxy.name) // 输出: Getting property "name"，然后输出: John

// 通过代理对象修改属性
proxy.age = 35 // 输出: Setting property "age" to "35"
console.log(proxy.age) // 输出: Getting property "age"，然后输出: 35
```

在这个示例中，我们创建了一个目标对象 `target`，然后定义了一个处理程序 `handler`，其中包含了 `get` 和 `set` 方法，用于拦截目标对象的属性访问和赋值操作。最后使用 `new Proxy(target, handler)` 创建了代理对象 `proxy`，并通过代理对象访问和修改了目标对象的属性。

使用 Proxy 可以实现更灵活和强大的对象操作，比如实现数据验证、数据绑定、数据跟踪等功能。它是 JavaScript 中的一个重要特性，可以帮助开发者更好地控制和管理对象的行为。

## Set Map

在 JavaScript 中，Set 和 Map 是两种不同的集合类型，它们提供了高效的数据存储和检索机制，用于处理不重复的值或键值对。

### Set 的用法和使用场景：

Set 是一种集合，它存储一组唯一的值，不允许重复。Set 中的值是唯一的，不会出现重复的元素。它的基本用法如下：

```javascript
const mySet = new Set()

mySet.add(1)
mySet.add(2)
mySet.add(3)
mySet.add(1) // 重复的值不会被添加

console.log(mySet) // 输出: Set { 1, 2, 3 }
```

常见的使用场景包括：

- 去重：可以用来去除数组中的重复元素。
- 标记：可以用来标记一组数据中是否存在某个值。
- 集合运算：可以进行集合的交集、并集、差集等操作。

### Map 的用法和使用场景：

Map 是一种键值对的集合，它存储键值对的关联关系，键是唯一的，每个键对应一个值。Map 中的键可以是任意类型的值，值也可以是任意类型的值。它的基本用法如下：

```javascript
const myMap = new Map()

myMap.set('name', 'John')
myMap.set('age', 30)

console.log(myMap.get('name')) // 输出: John
console.log(myMap.get('age')) // 输出: 30
```

常见的使用场景包括：

- 键值对存储：用来存储键值对数据，比如存储对象的属性和属性值。
- 缓存：可以用来存储计算结果，以提高性能。
- 数据映射：可以用来存储数据之间的映射关系，比如实现一种数据转换。

总的来说，Set 和 Map 是 JavaScript 中常用的集合类型，它们提供了方便的数据存储和检索机制，适用于各种不同的场景和需求。

### **Diff 算法**

React 是一个用于构建用户界面的 JavaScript 库。其核心思想是通过组件化的方式构建用户界面，将界面拆分为独立的组件，并通过组件之间的数据流来管理状态和交互。以下是 React 的基本原理：

1. **虚拟 DOM：** 使用 JS 对象来描述真实的 DOM 结构，使用虚拟 DOM 有如下的优点：

   > - 相较于 DOM 的体积和速度优势
   > - 虚拟 DOM 发挥优势的时机主要体现在更新的时候，相比较 innerHTML 要将已有的 DOM 节点全部销毁，虚拟 DOM 能够做到针对 DOM 节点做最小程度的修改
   > - 多平台渲染的抽象能力
   > - 浏览器、Node.js 宿主环境使用 ReactDOM 包
   > - Native 宿主环境使用 ReactNative 包

2. **组件化：** React 将用户界面拆分为独立的组件，每个组件负责管理自己的状态和 UI 渲染。组件可以是函数组件（Functional Components）或类组件（Class Components），它们都可以接收输入（props）并返回 UI 元素。

   > 将**可变部分**与**不可变部分**进行分离，**使不变的部分能够命中 bailout 策略**从而优化组件的性能

3. **声明式编程：** React 使用声明式编程的方式描述用户界面。开发者只需要定义界面应该是什么样子，而不需要关心如何实现。React 负责根据当前状态和数据来自动更新界面。

4. **状态管理：** React 使用状态（State）来表示组件的内部数据和状态。状态可以通过 `useState`、`useReducer` 等 Hook 函数来创建和管理。当状态发生变化时，React 会重新渲染相应的组件，并更新页面。

5. **状态管理：** React 使用单向数据流来管理组件之间的数据传递和通信。数据从父组件传递到子组件，子组件通过回调函数将数据更新传递给父组件。这种单向数据流的设计使得数据流动更加可控和可预测。

6. **Diff 算法：**diff 计算发生在更新阶段，当第一次渲染完成后，就会产生 Fiber 树，再次渲染的时候（更新），就会拿新的 JSX 对象（vdom）和旧的 FiberNode 节点进行一个对比，再决定如何来产生新的 FiberNode，它的目标是尽可能的复用已有的 Fiber 节点。

   React 不使用双端 diff 的原因：

   > 由于双端 diff 需要向前查找节点，但每个 FiberNode 节点上都没有反向指针，即前一个 FiberNode 通过 sibling 属性指向后一个 FiberNode，只能从前往后遍历，而不能反过来，因此该算法无法通过双端搜索来进行优化

   > diff 分为单节点 diff 和多节点 diff复用
   >
   > **单节点diff ：**
   >
   > 1. 判断 key 是否相同
   > 2. 如果 key 相同，再判断 type 是否相同
   >
   > **多节点 diff 会分为两轮遍历：**
   >
   > 第一轮遍历会从前往后进行遍历，存在以下三种情况：
   >
   > - 如果新旧子节点的key 和 type 都相同，说明可以复用
   > - 如果新旧子节点的 key 相同，但是 type 不相同，这个时候就会根据 ReactElement 来生成一个全新的 fiber，旧的 fiber 被放入到 deletions 数组里面，回头统一删除。但是注意，此时遍历并不会终止
   > - 如果新旧子节点的 key 和 type 都不相同，结束遍历
   >
   > 如果第一轮遍历被提前终止了，那么意味着还有新的 JSX 元素或者旧的 FiberNode 没有被遍历，因此会采用第二轮遍历去处理。
   >
   > 第二轮遍历会遇到三种情况：
   >
   > - 只剩下旧子节点：将旧的子节点添加到 deletions 数组里面直接删除掉（删除的情况）
   > - 只剩下新的 JSX 元素：根据 ReactElement 元素来创建 FiberNode 节点（新增的情况）
   > - 新旧子节点都有剩余：会将剩余的 FiberNode 节点放入一个 map 里面，遍历剩余的新的 JSX 元素，然后从 map 中去寻找能够复用的 FiberNode 节点，如果能够找到，就拿来复用。（移动的情况）

Fiber 将渲染过程分解成小的、可中断的工作单元（时间切片），通过分解渲染过程、动态调度任务的执行顺序和优先级，使得 React 应用能够更好地响应用户操作和浏览器事件从而提高了渲染的效率和可响应性。

> 实现方式：
>
> 1.  **Fiber 数据结构：** Fiber 是 React 中用于表示虚拟 DOM 节点的数据结构。它包含了描述当前节点的信息，例如节点类型、子节点、父节点等。Fiber 数据结构的设计使得 React 能够更灵活地处理渲染过程，并支持增量渲染和优先级调度。
> 2.  **调度器（Scheduler）：** 调度器负责协调和调度渲染工作，决定哪些任务应该先执行。它根据任务的优先级和当前的渲染阶段来动态地调整任务的执行顺序，以确保用户能够尽快看到重要的内容。
> 3.  **协调器（Reconciler）：** 协调器负责实现虚拟 DOM 的协调和更新。它使用 Fiber 数据结构来表示和管理虚拟 DOM 树，并根据变更情况进行增量更新。协调器会根据更新的优先级和类型，生成对应的 Fiber 节点，并构建 Fiber 树。
> 4.  **调度优先级策略：** React Fiber 使用了一种基于优先级的调度策略，以确保重要的任务能够优先执行。调度器会根据任务的优先级和当前的渲染阶段，动态地调整任务的执行顺序。例如，它会优先执行用户交互相关的任务，以提高用户体验。
> 5.  **渲染阶段（Phases）：** React Fiber 将渲染过程分解成多个阶段，每个阶段处理特定类型的任务。这些阶段包括协调阶段（Reconciliation Phase）、布局阶段（Layout Phase）、绘制阶段（Paint Phase）等。这种分解可以让 React 在多个渲染周期中交错处理不同的任务，提高渲染的效率和响应性。
> 6.  **错误处理机制：** React Fiber 引入了错误边界的概念，使得 React 应用能够更好地处理运行时错误。错误边界允许组件捕获和处理其子组件中发生的错误，从而避免整个应用的崩溃。
>
> - Scheduler（调度器）：调度任务的优先级，高优先级任务会优先进入到 Reconciler
> - Reconciler（协调器）：VDOM 的实现，负责根据自变量变化计算出 UI 变化
> - Renderer（渲染器）：负责将 UI 变化渲染到宿主环境中
>
> 首先引入了 Fiber 的概念，通过一个对象来描述一个 DOM 节点，但是和之前方案不同的地方在于，每个 Fiber 对象之间通过链表的方式来进行串联。通过 child 来指向子元素，通过 sibling 指向兄弟元素，通过 return 来指向父元素。
>
> 在新架构中，Reconciler 协调器 中的更新流程从递归变为了“可中断的循环过程”。每次循环都会调用 shouldYield 判断当前的 TimeSlice 是否有剩余时间，没有剩余时间则暂停更新流程，将主线程还给渲染流水线，等待下一个宏任务再继续执行。这样就解决了 CPU 的瓶颈问题。
>
> 另外在新架构中还引入了 Scheduler 调度器，用来调度任务的优先级，从而解决了 I/O 的瓶颈问题。

### 渲染阶段（Render Phase）

### 提交阶段（Commit Phase）

在 React 中，渲染（Render）阶段和提交（Commit）阶段是 React 更新过程中的两个重要阶段，它们分别负责不同的任务，如下所示：

1. **渲染阶段（Render Phase）：**
   - **Schedule（调度）：** 在渲染阶段的开始，React 会调度更新任务。调度器会根据任务的优先级和其他条件来确定任务的执行顺序，并将任务添加到任务队列中等待执行。
   - **Reconcile（协调）：** 在调度完成后，React 开始执行协调阶段。协调阶段主要负责协调新旧虚拟 DOM 树之间的差异，并生成更新操作。React 使用 Diff 算法比较新旧虚拟 DOM 树的差异，并将差异转化为一系列更新操作，以便后续提交阶段应用。

2. **提交阶段（Commit Phase）：**
   - **Commit（提交）：** 在协调阶段完成后，React 开始执行提交阶段。提交阶段主要负责将更新操作应用到真实 DOM 中，并完成页面的更新。React 会按照更新操作的顺序，从根节点开始逐步应用更新，直到所有更新都被应用到真实 DOM 中。提交阶段还包括一些其他任务，如执行生命周期方法、触发副作用等。

总的来说，渲染阶段负责调度更新任务和生成更新操作，而提交阶段负责将更新操作应用到真实 DOM 中，并完成页面的更新。这两个阶段相互配合，共同完成 React 更新过程中的任务，从而实现页面的更新和组件的渲染。

## React 中的事件处理

React 中的事件处理与普通的 JavaScript 事件处理有所不同，主要是因为 React 使用了合成事件（Synthetic Event）来封装浏览器原生事件，并提供了一种更一致和更高效的事件处理方式。以下是 React 中事件处理的基本流程：

1. **事件绑定：** 在 JSX 中，可以通过将事件处理函数作为属性传递给元素来绑定事件，例如 `onClick`、`onMouseDown` 等。事件处理函数通常是组件的方法。

   ```js
   <button onClick={this.handleClick}>Click me</button>
   ```

2. **事件处理函数：** 事件处理函数是组件的方法，通常在组件的定义中定义。它们接收一个合成事件对象作为参数，可以在函数中访问事件相关的信息，例如事件类型、目标元素等。

   ```js
   handleClick(event) {
     console.log('Button clicked');
   }
   ```

3. **合成事件：** React 使用合成事件来封装原生事件，使得事件处理过程更一致和可预测。合成事件提供了与原生事件相似的接口，但实际上是跨浏览器兼容的。

4. **事件委托：** React 使用事件委托来管理事件处理，即事件处理函数被绑定到顶层容器上，而不是直接绑定到每个元素上。当事件发生时，React 会根据事件的目标元素冒泡过程来确定要调用哪个事件处理函数。

5. **事件池：** React 使用了事件池来管理合成事件对象的创建和回收，以减少内存消耗和提高性能。当事件处理函数执行完成后，合成事件对象会被重用，而不是立即被销毁。

总的来说，React 的事件处理与普通的 JavaScript 事件处理类似，但使用了合成事件来封装原生事件，并提供了一种更一致和更高效的事件处理方式。通过事件绑定、事件处理函数、合成事件、事件委托和事件池等机制，React 能够实现灵活、高效和可预测的事件处理。

### useState 和 useReducer 有什么样的区别？

`useState` 和 `useReducer` 都是 React Hooks，用于在函数组件中管理状态。它们之间的主要区别在于状态管理的方式和适用场景：

###### useState

- `useState` 是 React 提供的基本 Hooks 之一，用于在函数组件中添加局部状态。
- `useState` 返回一个包含当前状态值和更新状态值的数组，通常使用数组解构来获取状态值和更新函数。
- 使用 `useState` 时，每个状态都独立存在，状态更新不会影响其他状态，适用于简单的状态管理场景。
- 通常用于管理单个或少量的简单状态，如开关状态、输入框的值等。

```js
const [count, setCount] = useState(0)
```

###### useReducer

- `useReducer` 也是 React 提供的 Hooks 之一，用于在函数组件中进行复杂的状态管理。
- `useReducer` 接收一个包含状态和操作状态的 reducer 函数，以及初始状态作为参数，返回当前状态值和派发动作的 dispatch 函数。
- 使用 `useReducer` 时，多个相关状态可以被组合在一起，并由一个 reducer 函数进行管理，状态之间可以有依赖关系，适用于复杂的状态逻辑和状态共享场景。
- 通常用于管理多个相关联的状态或具有复杂更新逻辑的状态，如表单状态、动态列表等。

```js
import React, { useReducer } from 'react'

// 定义 reducer 函数
const reducer = (state, action) => {
  switch (action.type) {
    case 'increment':
      return { count: state.count + 1 }
    case 'decrement':
      return { count: state.count - 1 }
    default:
      return state
  }
}

const Counter = () => {
  // 初始化状态和 dispatch 函数
  const initialState = { count: 0 }
  const [state, dispatch] = useReducer(reducer, initialState)
  return (
    <div>
      <h1>Count: {state.count}</h1>
      <button onClick={() => dispatch({ type: 'increment' })}>Increment</button>
      <button onClick={() => dispatch({ type: 'decrement' })}>Decrement</button>
    </div>
  )
}
export default Counter
```

总的来说，`useState` 和 `useReducer` 都是用于管理状态的 React Hooks，但它们适用于不同的场景。`useState` 适用于简单的状态管理，而 `useReducer` 则适用于复杂的状态管理和状态共享场景。选择合适的 Hooks 取决于你的具体需求和项目的复杂度。

### effect相关hook

> 面试题：说一说 useEffect 和 useLayoutEffect 的区别？

###### useEffect

- `useEffect` 是在浏览器完成布局和绘制后才执行的，因此它不会阻塞浏览器渲染。它是异步执行的，不会阻塞浏览器渲染线程。
- `useEffect` 中的副作用代码可能在浏览器渲染完成后才执行，因此不会影响页面的更新和用户体验。
- 通常用于处理不需要立即执行的副作用，例如数据获取、订阅事件等。

###### useLayoutEffect

- `useLayoutEffect` 与 `useEffect` 类似，但它是在浏览器完成布局但尚未绘制页面之前执行的，即在浏览器执行绘制之前立即执行。
- `useLayoutEffect` 中的副作用代码会在浏览器更新屏幕之前同步执行，因此可能会阻塞浏览器的渲染线程，导致页面的阻塞或闪烁。
- 通常用于需要立即执行的副作用，例如在渲染之前进行 DOM 操作、测量元素大小等。

```js
import React, { useState, useEffect } from 'react'

const Example = () => {
  const [count, setCount] = useState(0)

  // useEffect 接收一个函数作为参数，在每次渲染后执行
  useEffect(() => {
    // 执行副作用操作，例如订阅事件、数据获取、手动操作 DOM 等
    document.title = `You clicked ${count} times`

    // 返回一个清除函数，用于清理副作用
    return () => {
      document.title = 'React App' // 在组件卸载时重置标题
    }
  }, [count]) // 可选参数，指定依赖项数组，只有依赖项发生变化时才会重新执行 useEffect

  return (
    <div>
      <p>You clicked {count} times</p>
      <button onClick={() => setCount(count + 1)}>Click me</button>
    </div>
  )
}

export default Example
```

### useCallback 和 useMemo 的区别是什么

`useCallback` 和 `useMemo` 都是 React 提供的 Hooks，用于性能优化和避免不必要的重新渲染。它们的主要区别在于其用途和返回值：

###### **useCallback：**

- `useCallback` 用于缓存回调函数，以便在依赖未变时避免创建新的回调函数。
- `useCallback` 接收一个回调函数和依赖项数组作为参数，并返回一个 memoized 的回调函数。
- 当依赖项数组中的任何一个值发生变化时，`useCallback` 将返回一个新的回调函数，否则它将返回之前缓存的回调函数。
- 主要用于优化子组件的性能，避免不必要的函数重新创建和传递给子组件。

```jsx
const memoizedCallback = useCallback(() => {
  doSomething(a, b)
}, [a, b])
```

###### **useMemo：**

- `useMemo` 用于缓存计算值，以便在依赖未变时避免重复计算。
- `useMemo` 接收一个计算函数和依赖项数组作为参数，并返回一个 memoized 的值。
- 当依赖项数组中的任何一个值发生变化时，`useMemo` 将重新计算并返回一个新的值，否则它将返回之前缓存的值。
- 主要用于优化计算量较大或昂贵的值的计算过程，避免不必要的重复计算。

```jsx
const memoizedValue = useMemo(() => computeExpensiveValue(a, b), [a, b])
```

总的来说，`useCallback` 用于缓存回调函数，而 `useMemo` 用于缓存计算值。它们都是在性能优化方面非常有用的工具，可以帮助减少不必要的重新渲染和计算，提高应用的性能。选择使用哪一个取决于你需要优化的具体场景。

## Scheduler 任务调度

在 React 中，Scheduler 是一个用于调度任务的模块，它负责管理 React 应用中的异步任务的执行顺序和优先级。Scheduler 的设计目的是优化 React 应用的性能，提高页面的响应速度和流畅度。

Scheduler 主要包含以下几个核心概念：

1. **任务调度：** Scheduler 负责将任务按照优先级进行调度，以确保页面的重要任务优先执行。它使用一种称为“调度器”的机制来管理任务队列，并根据任务的优先级来确定下一个执行的任务。

2. **优先级策略：** Scheduler 使用优先级策略来确定任务的执行顺序。它将任务分为不同的优先级（如高优先级、中优先级、低优先级等），并根据任务的重要程度和紧急程度来确定任务的执行顺序。

3. **中断和暂停：** Scheduler 允许中断和暂停任务的执行，以便处理更紧急的任务或优化页面的性能。它提供了一些 API 来控制任务的执行，例如 `scheduleCallback`、`cancelCallback`、`flushSyncCallbackQueue` 等。

4. **任务超时和延迟：** Scheduler 允许设置任务的超时和延迟时间，以确保任务在规定的时间内完成。它提供了一些 API 来设置任务的延迟时间和超时时间，例如 `scheduleCallback`、`scheduleDelayedCallback` 等。

通过以上功能，Scheduler 可以帮助开发者更好地管理 React 应用中的异步任务，提高页面的响应速度和流畅度。Scheduler 的设计使得 React 应用能够更好地适应不同的环境和设备，并提供更好的用户体验。

## React性能优化

### 内部优化策略

> 在 React 内部，性能优化策略可以分为：
>
> - eagerState 策略
>   - eagerState 的核心逻辑是如果某个状态更新前后没有变化，则可以跳过后续的更新流程。该策略将状态的计算提前到了 schedule 阶段之前。当有 FiberNode 命中 eagerState 策略后，就不会再进入 schedule 阶段，直接使用上一次的状态。
>   - 该策略有一个前提条件，那就是当前的 FiberNode 不存在待执行的更新
>
> - **bailout 策略** (常用)
>
>   其本质就是将**可变部分**与**不可变部分**进行分离，**使不变的部分能够命中 bailout 策略从而优化组件的性能**
>
>   如果在比较过程中发现组件的 props、state 或上下文（context）没有发生变化，且组件的子树也没有发生变化，那么 React 就会放弃对该组件的重新渲染，这种放弃重新渲染的行为就称为 Bailout。
>   - 我们知道 beginWork 的作用主要是生成 wipFiberNode 的子 FiberNode，要达到这个目录存在两种方式：
>     - 通过 reconcile 流程生成子 FiberNode
>     - 通过命中 bailout 策略来复用子 FiberNode
>
>     在前面我们讲过，所有的变化都是由“自变量”的改变造成的，在 React 中自变量：
>     - state
>     - props
>     - context

### Suspense

Suspense 是 React 中的一个特性，用于处理组件异步加载过程中的等待状态。通过 Suspense，你可以在组件加载异步数据时显示一个自定义的加载指示器，以提高用户体验。Suspense 的主要作用是简化异步数据加载过程中的代码逻辑，使得代码更加清晰和易于理解。

### startTransition

`startTransition` 函数接受一个回调函数作为参数，并在回调函数中执行可能会造成阻塞的更新操作。

`startTransition` 主要适用于以下几种场景：

1. **数据加载：** 当需要加载大量数据或执行复杂的数据处理逻辑时，这些操作可能会阻塞用户界面的渲染。使用 `startTransition` 可以将这些数据加载或处理操作放在后台执行，以保持页面的响应性和流畅性。

2. **优化复杂更新：** 当需要进行复杂的页面更新操作，可能会导致一些页面元素的频繁重绘，从而影响用户体验。使用 `startTransition` 可以将这些更新操作放在后台执行，以减少页面的重绘次数，提高页面的性能和响应速度。

3. **优化用户交互：** 当用户执行一些可能会触发大量更新的交互操作时，比如拖动、滚动或输入等，这些操作可能会导致页面的重新渲染，从而影响用户体验。使用 `startTransition` 可以将这些更新操作放在后台执行，以保持页面的流畅性。

### useDeferredValue

`useDeferredValue` 是 React 中的一个钩子函数，用于延迟更新某个值，以提高用户体验。它通常与 `startTransition` 函数一起使用，用于延迟更新某个状态，直到渲染过渡完成后再显示更新后的值。

具体来说，`useDeferredValue` 可以将某个值标记为“延迟值”，并返回一个经过延迟的值。当这个值发生变化时，React 不会立即更新组件，而是等待渲染过渡完成后再更新组件。这样可以避免在渲染过渡过程中频繁更新组件，从而提高用户体验。

`useDeferredValue` 的主要场景包括：

> 1. **输入框实时搜索：** 当用户在输入框中输入内容时，通常会触发搜索功能，并实时显示搜索结果。使用 `useDeferredValue` 可以延迟更新搜索关键词，以减少频繁的搜索请求，提高性能和用户体验。
> 2. **滚动加载列表：** 在滚动加载列表的场景中，通常需要根据用户滚动的位置动态加载列表项。使用 `useDeferredValue` 可以延迟更新滚动位置，以避免频繁触发滚动加载操作，从而提高性能和用户体验。
> 3. **图像或视频加载：** 当页面中包含大量的图像或视频资源时，可能会导致页面加载速度变慢。使用 `useDeferredValue` 可以延迟加载图像或视频资源，以减少页面加载时间，提高性能和用户体验。
> 4. **优化复杂更新：** 当需要进行复杂的页面更新操作时，可能会导致页面的频繁重绘，影响用户体验。使用 `useDeferredValue` 可以延迟更新操作，以减少页面的重绘次数，提高性能和用户体验。

总的来说，`useDeferredValue` 主要用于优化页面的性能和用户体验，在一些需要延迟更新的场景中能够发挥很好的作用。通过延迟更新某些值，可以确保页面的流畅性和响应性，提高用户体验。

## 状态管理

### Redux

Redux 的工作流程可以概括为以下几个步骤：

> 1. **Action 的派发：** 在应用中的某个地方（比如用户交互、网络请求响应等），会触发一个 Action，这个 Action 是一个包含描述事件的信息的普通 JavaScript 对象，通常包括一个 `type` 字段来描述事件的类型，以及一些其他的字段来携带额外的数据。
> 2. **Reducer 的执行：** 一旦有 Action 被派发，Redux 会将这个 Action 传递给 Reducer。Reducer 是一个纯函数，它接收当前的状态（State）和被派发的 Action，然后根据 Action 的类型来生成新的状态。Reducer 返回的新状态会替换掉当前的状态，因此 Redux 的状态是不可变的。
> 3. **更新 Store：** 当新状态被生成后，Redux 会将其存储在一个称为 Store 的容器中。Store 包含了整个应用的状态树，并且提供了一些方法来获取状态、派发 Action 和注册监听器等功能。
> 4. **通知订阅者：** 如果有组件订阅了 Store 中的某些状态，那么一旦状态发生变化，Redux 会通知这些订阅者，让它们可以获取到最新的状态并更新自己。

### zustand

Zustand 是一个简单而强大的状态管理库，它使用了 React 的 Context API 来管理状态，并通过一种函数式的方式来定义和更新状态。下面是 Zustand 的工作流程：

1. **创建状态容器：** 首先，你需要使用 `create` 函数来创建一个状态容器。这个函数接受一个回调函数作为参数，在这个回调函数中定义了状态和更新状态的方法。例如：

```javascript
import create from 'zustand'

const useStore = create((set) => ({
  count: 0,
  increment: () => set((state) => ({ count: state.count + 1 })),
  decrement: () => set((state) => ({ count: state.count - 1 })),
}))
```

2. **使用状态容器：** 一旦创建了状态容器，你就可以在组件中使用 `useStore` 钩子函数来获取状态和更新状态的方法，并将它们应用到你的组件中。例如：

```javascript
import React from 'react'
import { useStore } from './useStore'

function Counter() {
  const { count, increment, decrement } = useStore()

  return (
    <div>
      <h1>Count: {count}</h1>
      <button onClick={increment}>Increment</button>
      <button onClick={decrement}>Decrement</button>
    </div>
  )
}
```

在这个例子中，我们使用了 `useStore` 钩子函数来获取状态 `count` 和更新状态的方法 `increment` 和 `decrement`，并将它们应用到计数器组件中。

3. **更新状态：** 当用户与你的应用交互时，你可以调用状态容器中的更新方法来更新状态。Zustand 使用内部的 `set` 函数来更新状态，你可以在更新方法中使用它来更新状态。例如：

```javascript
const useStore = create((set) => ({
  count: 0,
  increment: () => set((state) => ({ count: state.count + 1 })),
  decrement: () => set((state) => ({ count: state.count - 1 })),
}))
```

在这个例子中，我们定义了 `increment` 和 `decrement` 方法来更新状态 `count`，并在这些方法中使用 `set` 函数来更新状态。

# Vite

## 为什么说 vite 比 webpack 要快

和 `webpack` 对比，为什么 `vite` 的冷启动、热启动、热更新都会快？这就要说说二者的区别。

使用 `webpack` 时，从 `yarn start` 命令启动，到最后页面展示，需要经历的过程：

1. 以 `entry` 配置项为起点，做一个全量的打包，并生成一个入口文件 `index.html` 文件；
2. 启动一个 `node` 服务；
3. 打开浏览器，去访问入 `index.html`，然后去加载已经打包好的 `js`、`css` 文件；

使用 `vite` 时， 从 `vite` 命令启动，到最后的页面展示，需要经历的过程：

1. 使用 [esbuild](https://link.juejin.cn?target=https%3A%2F%2Fesbuild.github.io%2F) [预构建依赖](https://link.juejin.cn?target=https%3A%2F%2Fcn.vitejs.dev%2Fguide%2Fdep-pre-bundling.html)，提前将项目的第三方依赖格式化为 `ESM` 模块；
2. 启动一个 `node` 服务；
3. 打开浏览器，去访问 `index.html`；
4. 基于浏览器已经支持原生的 `ESM` 模块, 逐步去加载入口文件以及入口文件的依赖模块。浏览器发起请求以后，`dev server` 端会通过 `middlewares` 对请求做拦截，然后对源文件做 `resolve`、`load`、`transform`、`parse` 操作，然后再将转换以后的内容发送给浏览器。

> 开发模式下 `vite` 比 `webpack` 快的原因：
>
> 1. `vite` 不需要做全量的打包，这是比 `webpack` 要快的最主要的原因；
> 2. `vite` 在解析模块依赖关系时，利用了 `esbuild`，更快（esbuild 使用 Go 编写，并且比以 JavaScript 编写的打包器预构建依赖快 10-100 倍）；
> 3. `按需加载`；模块之间的依赖关系的解析由浏览器实现。Vite 只需要在浏览器请求源码时进行转换并按需提供源码。根据情景动态导入代码，即只在当前屏幕上实际使用时才会被处理。
> 4. `充分利用缓存`；Vite 利用 HTTP 头来加速整个页面的重新加载（再次让浏览器为我们做更多事情）：源码模块的请求会根据 `304 Not Modified` 进行协商缓存，而依赖模块请求则会通过 `Cache-Control: max-age=31536000,immutable` 进行强缓存，因此一旦被缓存它们将不需要再次请求。

## 转换树结构

```js
//filter map   filter过滤出父级节点  对父节点使用重复调用arrayToTree方法找到其子节点
function arrayToTree(array, parentId) {
  return array
    .filter((item) => item.parentId === parentId)
    .map((item) => ({
      ...item,
      children: arrayToTree(array, item.id),
    }))
}

// 对象键存储所有数据
function flatToTree(flatArray, parentId) {
  const map = {}
  const tree = []
  // 将节点按照 id 作为键存储到 map 中
  flatArray.forEach((node) => {
    map[node.id] = { ...node, children: [] }
  })
  // 将节点连接起来
  flatArray.forEach((node) => {
    if (node.parentId !== parentId) {
      map[node.parentId].children.push(map[node.id])
    } else {
      tree.push(map[node.id])
    }
  })

  return tree
}

//递归
function flatToTreeRecursive(flatArray, parentId = null) {
  const tree = []
  flatArray.forEach((node) => {
    if (node.parentId === parentId) {
      const children = flatToTreeRecursive(flatArray, node.id)
      if (children.length) {
        node.children = children
      }
      tree.push(node)
    }
  })

  return tree
}
```
