# React 18 与 React 19

为了避免概念打架，我们先统一一下术语：

- 局部状态：只在当前组件或很小范围内使用
- 跨层共享状态：需要在一棵组件树里传递给很多后代
- 全局客户端状态：跨页面、跨区域、跨组件复用的业务状态或 UI 状态
- 服务端状态：从后端获取、需要缓存和重新验证的数据

这篇文章会先讲客户端状态管理主线，再补一整块服务端状态管理，帮助你把两类问题彻底分开。

## React 18 中的状态管理

React 18 的原生状态管理主线非常清晰：简单状态用 `useState`，复杂更新逻辑用 `useReducer`，跨层共享用 `Context + useContext`。很多中小项目只靠这三件套就已经够用了。

### useState：处理局部简单状态

它解决什么问题：当前组件内部少量、直接、更新路径清晰的状态。

最典型的场景包括：

- 输入框内容
- 弹窗开关
- tabs 当前项
- loading 布尔值
- 当前页码

```tsx
import { useState } from 'react'

export function SearchBox() {
  const [keyword, setKeyword] = useState('')

  return (
    <input
      value={keyword}
      onChange={(event) => setKeyword(event.target.value)}
      placeholder="搜索关键词"
    />
  )
}
```

什么时候优先选它：

- 状态只在一个组件里用
- 更新逻辑非常直接
- 不需要抽象 action

注意点：如果你已经开始维护多个互相关联字段，或者更新逻辑充满 `if/else`，那就该考虑 `useReducer` 了。

### useReducer：处理复杂更新逻辑

它解决什么问题：状态字段彼此关联、更新规则明确、事件类型开始变多时，把更新逻辑收敛到一个 reducer。

```tsx
import { useReducer } from 'react'

type FilterState = {
  keyword: string
  sortBy: 'latest' | 'price'
  onlyInStock: boolean
}

type FilterAction =
  | { type: 'setKeyword'; payload: string }
  | { type: 'setSortBy'; payload: 'latest' | 'price' }
  | { type: 'toggleStock' }
  | { type: 'reset' }

const initialState: FilterState = {
  keyword: '',
  sortBy: 'latest',
  onlyInStock: false,
}

function filterReducer(state: FilterState, action: FilterAction): FilterState {
  switch (action.type) {
    case 'setKeyword':
      return { ...state, keyword: action.payload }
    case 'setSortBy':
      return { ...state, sortBy: action.payload }
    case 'toggleStock':
      return { ...state, onlyInStock: !state.onlyInStock }
    case 'reset':
      return initialState
    default:
      return state
  }
}

export function FilterPanel() {
  const [state, dispatch] = useReducer(filterReducer, initialState)

  return (
    <div>
      <input
        value={state.keyword}
        onChange={(event) => dispatch({ type: 'setKeyword', payload: event.target.value })}
      />

      <button onClick={() => dispatch({ type: 'setSortBy', payload: 'price' })}>按价格排序</button>

      <button onClick={() => dispatch({ type: 'toggleStock' })}>
        {state.onlyInStock ? '仅看有货' : '全部商品'}
      </button>
    </div>
  )
}
```

什么时候优先选它：

- 状态字段很多，而且互相关联
- 更新动作可以被明确建模成 action
- 你希望逻辑更集中、更容易测试和维护

注意点：`useReducer` 适合复杂逻辑，但不代表它天然就是全局状态管理方案。它首先还是组件级状态工具。

### Context + useContext：处理跨层共享

它解决什么问题：某个值需要被很多层后代使用，不想一层层手动传 props。

```tsx
import { createContext, useContext, useState } from 'react'

type Theme = 'light' | 'dark'

const ThemeContext = createContext<{
  theme: Theme
  setTheme: (theme: Theme) => void
} | null>(null)

function ThemeSwitcher() {
  const context = useContext(ThemeContext)

  if (!context) {
    throw new Error('ThemeSwitcher must be used within ThemeContext.Provider')
  }

  return (
    <button onClick={() => context.setTheme(context.theme === 'light' ? 'dark' : 'light')}>
      当前主题：{context.theme}
    </button>
  )
}

export function ThemeProviderExample() {
  const [theme, setTheme] = useState<Theme>('light')

  return (
    <ThemeContext.Provider value={{ theme, setTheme }}>
      <ThemeSwitcher />
    </ThemeContext.Provider>
  )
}
```

它特别适合：

- 主题
- 语言
- 当前用户信息
- 权限信息
- 某个区域内的共享配置

注意点：Context 是“共享通道”，不是完整状态库。它可以承载状态，但不负责组织复杂更新逻辑。

### useReducer + Context：React 18 常见原生全局状态组合

在 React 18 项目里，很多人会把 `useReducer` 和 Context 组合起来，做一个不依赖第三方库的轻量全局状态方案。

```tsx
import { createContext, useContext, useReducer } from 'react'

type AuthState = {
  userName: string
  isLoggedIn: boolean
}

type AuthAction = { type: 'login'; payload: string } | { type: 'logout' }

const initialAuthState: AuthState = {
  userName: '',
  isLoggedIn: false,
}

function authReducer(state: AuthState, action: AuthAction): AuthState {
  switch (action.type) {
    case 'login':
      return { userName: action.payload, isLoggedIn: true }
    case 'logout':
      return initialAuthState
    default:
      return state
  }
}

const AuthContext = createContext<{
  state: AuthState
  dispatch: React.Dispatch<AuthAction>
} | null>(null)

export function AuthProvider({ children }: { children: React.ReactNode }) {
  const [state, dispatch] = useReducer(authReducer, initialAuthState)

  return <AuthContext.Provider value={{ state, dispatch }}>{children}</AuthContext.Provider>
}

export function useAuth() {
  const context = useContext(AuthContext)
  if (!context) {
    throw new Error('useAuth must be used within AuthProvider')
  }
  return context
}
```

这套组合在 React 18 里很常见，因为它能同时满足：

- 有状态
- 有 action
- 能跨层共享
- 不需要额外安装状态库

但它也很快会碰到边界。

### React 18 原生方案的适用边界

React 18 的原生方案在小到中等复杂度项目里通常够用，但状态共享一旦变复杂，就会开始暴露成本：

- Provider 越来越多，组件树外层被包很多层
- Context value 一更新，依赖它的消费者可能会一起重渲染
- 多个业务域塞进同一个 Context 后，边界变模糊
- 组件树耦合越来越重，状态组织不再轻松

一句话总结：React 18 原生方案够强，但当共享状态跨页面、跨区域、跨很多组件反复复用时，组织成本会上升。

## React 19 中的状态管理

React 19 并没有把原生状态管理彻底重写，`useState`、`useReducer` 仍然是主线。它更像是在原有思路上做了语法和能力上的优化，尤其是 Context Provider 的写法更顺了。

### useState / useReducer 主线不变

React 19 并没有改变这两者的职责边界：

- 简单局部状态仍然优先 `useState`
- 复杂更新逻辑仍然优先 `useReducer`

也就是说，如果你在 React 18 里已经形成了清晰判断，升级到 React 19 后，这部分思路几乎不用改。

### Context Provider 新写法

React 18 里常见写法是：

```tsx
<ThemeContext.Provider value={theme}>
  <Page />
</ThemeContext.Provider>
```

React 19 开始可以直接写成：

```tsx
import { createContext, useState } from 'react'

const ThemeContext = createContext('light')

export function ThemeProviderExample() {
  const [theme, setTheme] = useState('light')

  return (
    <ThemeContext value={theme}>
      <button onClick={() => setTheme(theme === 'light' ? 'dark' : 'light')}>切换主题</button>
    </ThemeContext>
  )
}
```

这个变化的意义在于：

- 语法更简洁
- Context 本身更像一个可直接使用的 Provider 组件
- React 19 里新的 Context 写法更统一

但要注意，这只是 Provider 写法更顺了，不代表 Context 的组织问题消失了。

### `SomeContext.Provider` 在 React 19 中是什么位置

React 19 官方文档里，`SomeContext.Provider` 已经被视为 React 19 之前的旧写法对照。也就是说：

- React 18：`<SomeContext.Provider value={...}>`
- React 19：`<SomeContext value={...}>`

如果你维护的是 React 18 老项目，旧写法完全正常；如果是 React 19 新代码，优先直接用新的 Provider 形式更自然。

### `use(Context)` 作为补充能力

React 19 还引入了 `use(Context)` 这种读取 Context 的方式，但在大多数普通业务代码里，`useContext` 依然是更容易理解的主线。

你可以简单理解为：

- `useContext(Context)`：主流、稳定、最容易理解
- `use(Context)`：更灵活，但不是本文要展开的主要状态管理路线

所以在状态管理这篇文章里，我们只把它作为补充说明，不让它抢走主线。

### React 19 的结论

React 19 让原生状态共享的语法更顺，但没有改变原生方案的根本边界：

- 局部状态和中等复杂度状态，原生方案仍然很好用
- 一旦是复杂共享状态、跨页面状态、频繁局部读取的全局状态，原生方案的组织成本依旧会出现

也正因为这个边界还在，实际项目里我们才会继续引出 Zustand。

## Zustand

### 5.1 先判断这份状态该不该进 Zustand

第一步不是写 store，而是先做选择题。

可以先用下面这个判断规则：

| 场景                                                 | 优先方案     |
| ---------------------------------------------------- | ------------ |
| 组件内部独享状态                                     | `useState`   |
| 状态逻辑复杂但仍局部                                 | `useReducer` |
| 同一片树内共享且范围有限                             | `Context`    |
| 跨页面、跨区域、跨组件层级复用，且读写分散或更新频繁 | `Zustand`    |

适合进 Zustand 的常见状态：

- 搜索筛选条件
- 排序方式
- 当前选中的 tab / 面板
- 用户 UI 偏好
- 表单草稿
- 客户端缓存的临时业务状态

不建议一上来就进 Zustand 的状态：

- 单个输入框值
- 只在一个弹窗里使用的开关
- 只属于某一个组件的临时 hover 状态

第一条原则很简单：能局部解决，就先局部解决。

### 5.2 设计 store 的边界

确定要用 Zustand 后，下一步不是猛写字段，而是先定边界。

一个健康的 store，通常应该包含：

- `state`：真正要共享的数据
- `actions`：修改这些数据的函数
- 可选的派生读取方式：例如某些方便复用的选择逻辑

一个 store 不应该默认装下：

- 所有页面临时状态
- 明显属于服务端缓存的数据
- 与当前业务域无关的 UI 局部状态

推荐思路是“按业务域划分 store”，而不是先做一个巨大的全局 store。

例如：

- `useFilterStore`
- `useThemeStore`
- `useDraftStore`

而不是一开始就做一个：

```tsx
useAppStore
```

然后把所有东西都往里塞。

### 5.3 写第一个最小 store

这里用“过滤面板 + 排序 + 关键字”做示例

```tsx
import { create } from 'zustand'

type SortBy = 'latest' | 'price'

type FilterStore = {
  keyword: string
  sortBy: SortBy
  onlyInStock: boolean
  setKeyword: (keyword: string) => void
  setSortBy: (sortBy: SortBy) => void
  toggleOnlyInStock: () => void
  reset: () => void
}

const initialState = {
  keyword: '',
  sortBy: 'latest' as SortBy,
  onlyInStock: false,
}

export const useFilterStore = create<FilterStore>((set) => ({
  ...initialState,
  setKeyword: (keyword) => set({ keyword }),
  setSortBy: (sortBy) => set({ sortBy }),
  toggleOnlyInStock: () => set((state) => ({ onlyInStock: !state.onlyInStock })),
  reset: () => set(initialState),
}))
```

这段 store 已经包含了一个真实项目最小闭环：

- 共享状态字段
- 统一 action
- 重置能力

这里有一个很重要的习惯：  
把更新逻辑尽量收进 action，而不是让组件随手在外面拼一堆 `set(...)`。

### 5.4 在组件中读取状态

Zustand 最推荐的读取方式是 selector。

基础读法如下：

```tsx
const keyword = useFilterStore((state) => state.keyword)
const sortBy = useFilterStore((state) => state.sortBy)
```

为什么这是主线：

- 组件只订阅自己关心的那一小块状态
- 其他字段更新时，这个组件不容易被无关重渲染影响

完整示例：

```tsx
export function FilterSummary() {
  const keyword = useFilterStore((state) => state.keyword)
  const sortBy = useFilterStore((state) => state.sortBy)

  return (
    <p>
      关键词：{keyword || '无'} / 排序：{sortBy}
    </p>
  )
}
```

不推荐的默认写法是整 store 一把取出：

```tsx
const store = useFilterStore()
```

原因不是它绝对不能用，而是它很容易让组件订阅过多不需要的状态，后面会增加无关重渲染和维护成本。

### 5.5 在组件中更新状态

更新状态时，推荐直接调用 store action。

```tsx
export function FilterControls() {
  const keyword = useFilterStore((state) => state.keyword)
  const setKeyword = useFilterStore((state) => state.setKeyword)
  const setSortBy = useFilterStore((state) => state.setSortBy)
  const toggleOnlyInStock = useFilterStore((state) => state.toggleOnlyInStock)
  const reset = useFilterStore((state) => state.reset)

  return (
    <div className="flex flex-col gap-3">
      <input
        value={keyword}
        onChange={(event) => setKeyword(event.target.value)}
        placeholder="输入关键词"
      />

      <button onClick={() => setSortBy('price')}>按价格排序</button>
      <button onClick={toggleOnlyInStock}>切换库存筛选</button>
      <button onClick={reset}>重置全部条件</button>
    </div>
  )
}
```

这一节最重要的原则是：

- 组件负责触发动作
- store 负责组织更新逻辑

这样你后续如果要补额外规则，例如切换排序时清空分页，就只需要改 action，而不是到处翻组件代码。

### 5.6 处理异步 action

Zustand 并不限制你写异步 action。最常见的方式就是直接在 action 里写 `async/await`。

例如加载商品列表：

```tsx
import { create } from 'zustand'

type Product = {
  id: number
  name: string
}

type ProductStore = {
  items: Product[]
  loading: boolean
  error: string
  fetchProducts: () => Promise<void>
}

export const useProductStore = create<ProductStore>((set) => ({
  items: [],
  loading: false,
  error: '',
  fetchProducts: async () => {
    set({ loading: true, error: '' })

    try {
      const response = await fetch('/api/products')
      const data = (await response.json()) as Product[]
      set({ items: data, loading: false })
    } catch {
      set({ error: '加载失败', loading: false })
    }
  },
}))
```

组件里使用：

```tsx
import { useEffect } from 'react'

export function ProductList() {
  const items = useProductStore((state) => state.items)
  const loading = useProductStore((state) => state.loading)
  const error = useProductStore((state) => state.error)
  const fetchProducts = useProductStore((state) => state.fetchProducts)

  useEffect(() => {
    void fetchProducts()
  }, [fetchProducts])

  if (loading) return <p>加载中...</p>
  if (error) return <p>{error}</p>

  return (
    <ul>
      {items.map((item) => (
        <li key={item.id}>{item.name}</li>
      ))}
    </ul>
  )
}
```

这套模式很适合：

- 轻量异步流程
- 某个页面或业务域自己的请求状态
- 不需要复杂缓存策略的客户端数据流

但也要清楚边界：  
如果你的重点已经变成“请求缓存、去重、重新验证、失效策略”，那通常该考虑专门的服务端状态工具，而不是让 Zustand 一个人全做。

### 5.7 做持久化

一旦状态需要跨刷新保留，`persist` 会非常实用。

最常见的持久化场景包括：

- 用户主题
- 筛选条件
- 上次选择项
- 草稿

最小示例：

```tsx
import { create } from 'zustand'
import { persist } from 'zustand/middleware'

type PreferenceStore = {
  theme: 'light' | 'dark'
  keyword: string
  loading: boolean
  setTheme: (theme: 'light' | 'dark') => void
  setKeyword: (keyword: string) => void
  setLoading: (loading: boolean) => void
}

export const usePreferenceStore = create<PreferenceStore>()(
  persist(
    (set) => ({
      theme: 'light',
      keyword: '',
      loading: false,
      setTheme: (theme) => set({ theme }),
      setKeyword: (keyword) => set({ keyword }),
      setLoading: (loading) => set({ loading }),
    }),
    {
      name: 'preference-store',
      partialize: (state) => ({
        theme: state.theme,
        keyword: state.keyword,
      }),
    },
  ),
)
```

这里最值得注意的是 `partialize`。  
它的作用是：只把真正值得保留的字段写进存储。

适合持久化的：

- 主题
- 筛选条件
- 草稿
- 用户偏好

不适合默认持久化的：

- 短期 loading
- 临时错误态
- 明显依赖当前请求结果的数据

一句话判断：  
如果用户刷新页面后仍然希望这个值被保留，它才值得考虑持久化。

### 5.8 store 变大以后怎么拆

很多人第一次用 Zustand 很顺，第二个月就开始遇到问题：一个 store 越写越大，最后什么都在里面。

所以 store 变大以后，拆分是必修课。

什么时候该拆：

- 单个 store 的字段越来越多
- action 之间已经属于不同业务域
- 不同页面只用其中一部分状态
- 阅读 store 已经需要滚很久

最常见的拆法是按业务域拆：

- `useAuthStore`
- `useFilterStore`
- `useThemeStore`
- `useDraftStore`

如果你已经走到更复杂阶段，也可以继续用 slices pattern，把一个大业务域拆成多个 slice 再组合。但对大多数项目来说，先做到“按业务域拆 store”，就已经能解决 80% 的问题。

核心原则只有一个：  
不要为了“全局统一”而把所有状态塞进一个超大 store。

## 原生方案和 Zustand

| 维度         | React 原生方案             | Zustand                              |
| ------------ | -------------------------- | ------------------------------------ |
| 局部状态     | 很强，首选                 | 不推荐作为默认方案                   |
| 跨层共享     | Context 很方便             | 也能做，但通常没必要替代简单 Context |
| 复杂更新逻辑 | `useReducer` 足够清晰      | 适合跨组件共享后的统一组织           |
| 全局共享     | 可以实现，但组织成本会升高 | 更自然                               |
| 性能控制     | 需要小心 Context 更新扩散  | selector 更容易做按需订阅            |
| 心智负担     | 原生、简单、无需额外依赖   | 多一层 store 设计成本                |
| 适用项目规模 | 小到中型项目很好用         | 中型及以上共享状态更友好             |

最重要的一句话是：

Zustand 不是为了取代 `useState`，而是为了补足 React 原生方案在复杂共享状态上的组织成本。

## 怎么选状态管理方案

如果你只想记一个速查表，可以记下面这张：

| 场景                                      | 优先方案     |
| ----------------------------------------- | ------------ |
| 表单输入、开关、局部交互                  | `useState`   |
| 复杂表单流转、明确 action 模型            | `useReducer` |
| 主题、语言、权限、区域级共享配置          | `Context`    |
| 跨页面筛选、UI 偏好、草稿、客户端全局共享 | `Zustand`    |

最后的判断原则可以压缩成一句话：

小状态先原生，跨组件复杂共享再上 Zustand。
