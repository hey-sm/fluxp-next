# React Diff 算法与 Reconciliation 指南

React 在每次更新时，并不会简单地“整棵树重建再整页替换”。真正发生的是一套有约束的比较与复用过程：React 会根据新旧元素树的差异，判断哪些节点可以复用，哪些节点必须重建，以及哪些 DOM 变更需要在提交阶段真正落地。这个过程通常被开发者称为“Diff 算法”，在 React 官方语境中更准确的名称是 `reconciliation`。

本文的目标不是把 Diff 讲成抽象算法课，而是建立一套能指导工程实践的技术认知：React 为什么需要 Diff，它到底比较什么，`key` 为什么重要，组件状态为什么有时会保留、有时会被重置，以及这些规则如何与 Fiber、render phase、commit phase 连接起来。

## 什么是 React Diff

“Diff”在这里指的是 React 对两次渲染结果进行比较，从而决定最小必要更新的一套规则。更严格地说，React 不是在比较真实 DOM，而是在 `render phase` 中比较新旧 React 元素及其对应的 Fiber 结构，并为后续提交阶段准备需要执行的变更。

因此，React Diff 的核心问题不是“如何求出数学意义上的全局最优树编辑路径”，而是：

- 哪些节点可以继续复用
- 哪些节点必须销毁并重新创建
- 哪些属性、文本、子节点发生了变化
- 哪些组件的状态应该保留，哪些应该重置

React 官方历史文档明确说明，通用树编辑问题的最优算法复杂度很高，而 React 采用的是一套基于前提假设的启发式比较策略，以在工程上取得稳定且可接受的性能。来源：React 官方 `Reconciliation` 文档。  
链接：https://legacy.reactjs.org/docs/reconciliation.html

## 为什么 React 需要 Diff

如果每次状态变化都直接销毁整棵旧树并重新创建整棵新树，虽然实现简单，但代价会非常高：

- 大量本可复用的 DOM 节点被无意义重建
- 输入框、滚动位置、组件局部状态更容易丢失
- 更新粒度粗，提交成本高
- 大树更新时性能浪费明显

React 的目标并不是“绝不重新创建”，而是“在一套可预测规则下尽可能复用”。Diff 机制的意义就在于让 React 知道：

- 同一个位置上的节点是否还是“同一个东西”
- 如果还是同一个，就尽量复用并局部更新
- 如果不是同一个，就销毁旧子树并创建新子树

这也是 React 状态保留语义的基础。状态是否延续，并不是由变量名或组件文件名决定，而是由节点身份是否被 React 认为保持一致决定。

## React Diff 的基本前提

React 的 Diff 策略建立在两个非常重要的前提之上：

1. 不同类型的元素，会生成不同的树
2. 开发者可以通过 `key` 告诉 React 哪些子节点在多次渲染之间是稳定对应的

这两个前提直接决定了后续的大部分比较规则。如果没有这两条约束，React 很难在可接受的复杂度内完成稳定的树比较。

## React 比较的单位是什么

很多资料会把 React Diff 简化成“比较虚拟 DOM”。这个说法不算错误，但不够准确。更贴近现代 React 实现的理解是：

- JSX 会生成 React 元素
- React 在 render 过程中根据元素创建或复用 Fiber 节点
- Diff 发生在新旧元素及其对应 Fiber 的比较过程中
- 真实 DOM 更新发生在 commit phase

因此，Diff 更像一次“渲染期的结构判定”。它的产物不是立刻更新 DOM，而是给后续提交准备一组需要执行的变更。

## 节点比较的核心规则

React Diff 的核心不在于复杂公式，而在于少数几条稳定规则。

### 规则一：元素类型不同，整棵子树重建

如果同一位置上的新旧节点类型不同，React 会认为它们不是同一个节点，旧节点对应的整棵子树会被卸载，新节点及其子树会被重新创建。

例如：

```tsx
function App({ isArticle }: { isArticle: boolean }) {
  return isArticle ? (
    <article>
      <Panel />
    </article>
  ) : (
    <section>
      <Panel />
    </section>
  )
}
```

这里同一位置从 `article` 变成 `section`，对 React 来说，根宿主元素类型已经变化，因此这不是一次“局部修补”，而是一次节点替换。其下方对应子树也会重新进入创建流程。

对于组件也是同样的原则：

```tsx
function App({ loggedIn }: { loggedIn: boolean }) {
  return loggedIn ? <Dashboard /> : <LoginPage />
}
```

`Dashboard` 与 `LoginPage` 是不同类型的组件，React 不会尝试把它们当成同一组件做内部字段级别迁移，而是直接替换整棵子树。

### 规则二：元素类型相同，尝试复用节点并继续向下比较

如果同一位置上的节点类型相同，React 会优先复用已有节点，再比较其属性和子节点。

例如宿主元素：

```tsx
<div className="box" />
<div className="box active" />
```

这两次渲染的根节点类型都是 `div`，因此 React 倾向于复用原有 DOM 节点，只更新变化的属性。

组件也是同样的逻辑：

```tsx
<Profile userId={1} />
<Profile userId={2} />
```

因为类型仍然是 `Profile`，React 会复用该组件对应的节点身份，并以新的 props 重新执行组件，再继续比较其返回结果。

### 规则三：子节点默认按位置顺序比较

如果没有 `key` 提供额外身份信息，React 在一组兄弟节点之间主要依据位置顺序进行匹配。

例如：

```tsx
<>
  <li>A</li>
  <li>B</li>
</>
```

更新为：

```tsx
<>
  <li>X</li>
  <li>A</li>
  <li>B</li>
</>
```

如果没有稳定 `key`，React 会按顺序尝试匹配：

- 旧第 1 个和新第 1 个比较
- 旧第 2 个和新第 2 个比较
- 之后再处理新增节点

这正是列表头部插入场景下，错误使用索引 key 或不写 key 容易带来额外更新和状态错位的根源。

## key 在 Diff 中的作用

`key` 是 React 提供给开发者的身份提示。React 官方文档对这一点表述得非常明确：`key` 告诉 React 数组中的每一项分别对应谁，从而在插入、删除、排序时正确匹配前后两次渲染的节点。  
链接：https://react.dev/learn/rendering-lists

### key 解决的核心问题

在列表渲染中，如果只依赖位置，React 无法可靠判断“这是旧节点移动了位置”，还是“旧节点被删除并插入了一个全新节点”。`key` 的作用，就是让 React 在兄弟节点范围内建立稳定身份。

```tsx
function UserList({ users }: { users: { id: number; name: string }[] }) {
  return (
    <ul>
      {users.map((user) => (
        <li key={user.id}>{user.name}</li>
      ))}
    </ul>
  )
}
```

有了 `key={user.id}`，React 就能在更新时更可靠地识别：

- 哪个节点只是位置变了
- 哪个节点是新插入的
- 哪个节点被删除了

### key 不只是性能问题，也是状态身份问题

很多人把 `key` 只理解成“优化列表渲染性能”，这不完整。更重要的是，`key` 直接参与节点身份判定，因此会影响状态保留与重置。

React 官方在 `Preserving and Resetting State` 中明确说明：给节点设置不同的 `key`，会让 React 把它们视为不同的树，从而重建其状态与 DOM。  
链接：https://react.dev/learn/preserving-and-resetting-state

例如：

```tsx
function Messenger({ to }: { to: { id: number; name: string } }) {
  return <Chat key={to.id} contact={to} />
}
```

这里切换 `to.id` 时，React 会把不同联系人对应的 `<Chat>` 视为不同实例，从而重置输入框等局部状态。这不是副作用，而是 `key` 改变身份的直接结果。

### key 的使用规则

根据 React 官方文档，`key` 需要满足两个要求：

- 在同一组兄弟节点中唯一
- 在多次渲染之间保持稳定

不推荐的写法包括：

- `key={index}`，尤其在列表可能插入、删除、排序时
- `key={Math.random()}`，因为每次 render 都会变化

这些写法会破坏节点身份稳定性，进而带来状态错位、输入内容丢失和不必要的重建。

## 状态为什么会保留或重置

React 19 官方文档中，状态保留与重置已经不再被单独解释为“生命周期问题”，而是直接绑定到树中的位置与身份。这个视角比传统的“组件重渲染”说法更准确。

### 同一位置、同一类型、同一 key，状态通常会保留

如果一个组件在树中的身份没有变化，React 会尽量保留它对应的状态。

```tsx
function App({ userId }: { userId: number }) {
  return <Profile userId={userId} />
}
```

虽然 `userId` 变化了，但这里树中的节点仍然是同一位置上的 `Profile`，且没有更换 `key`，因此该组件自身状态默认会保留。

### 类型变化或 key 变化，状态会被重置

下面两种情况都会导致 React 认为节点身份发生变化：

- 节点类型变了
- 节点 `key` 变了

例如：

```tsx
function App({ mode }: { mode: 'view' | 'edit' }) {
  return mode === 'view' ? <Viewer /> : <Editor />
}
```

或者：

```tsx
function App({ formId }: { formId: string }) {
  return <Form key={formId} />
}
```

这两种情况都会触发重建，从而导致局部状态重置。

## 列表 Diff 的典型场景

理解列表更新，是掌握 React Diff 的关键。

### 尾部追加

尾部追加是最容易处理的场景。例如：

```tsx
['A', 'B'] -> ['A', 'B', 'C']
```

前两个节点可以稳定匹配，React 只需要处理末尾新增节点。

### 中间插入

中间插入或头部插入，如果没有稳定 `key`，React 只能更依赖位置匹配，容易导致后续节点都被当作“变化了”。

```tsx
['A', 'B'] -> ['X', 'A', 'B']
```

如果 key 不稳定，这种场景下不仅会带来额外更新，还可能导致列表项内部状态错位。

### 重排序

重排序是 `key` 价值最明显的场景之一。稳定 `key` 能帮助 React 理解“节点身份未变，只是位置调整了”，而不是把一连串节点都误判为被替换。

### 删除

列表删除场景下，React 需要知道究竟是哪个节点被移除。稳定 `key` 可以把“删除谁”从“按位置猜测”变成“按身份确认”。

## React 18 与 React 19 下的连续性

从开发者心智模型来看，React 18 与 React 19 并没有引入两套不同的 Diff 原理。节点比较、身份匹配、`key`、状态保留与重置这些基础规则，在现代 React 中保持连续。

需要准确表达的是：

- React 18 让现代渲染模型与并发调度成为主线
- React 19 在此基础上增强 Suspense、hydration、数据读取与开发体验
- 但 React 19 并没有重新定义“元素如何比较、节点如何复用、状态如何判定身份”这套基本规则

因此，学习 Diff 时，不应把它理解成“React 18/19 的版本差异点”，而应把它视为现代 React 渲染基础能力的一部分。

## Diff、Fiber 与 Render / Commit 的关系

Diff 不是一个脱离渲染流程独立存在的算法模块。它和 Fiber、render phase、commit phase 之间有明确分工。

### Diff 主要发生在 render phase

在 render 阶段，React 会重新执行受影响组件，并在新旧元素 / Fiber 之间做比较。这个过程会决定：

- 哪些 Fiber 可以复用
- 哪些节点需要打上更新标记
- 哪些子树需要删除或新建

换言之，Diff 发生在“计算下一次 UI”的阶段，而不是发生在 DOM 已经更新之后。

### commit phase 负责真正提交结果

Diff 的结果并不会立刻改动 DOM。真正的 DOM 创建、更新、删除，发生在 commit 阶段。

因此，可以把关系概括为：

- Fiber 提供可组织、可调度的节点结构
- Diff 负责在 render 中比较新旧树
- Commit 负责把比较结果真正落地到宿主环境

## 常见误区

### 误区一：React Diff 会计算全局最优 DOM 更新路径

React 采用的是启发式比较策略，而不是通用树编辑问题的最优解算法。它追求的是工程上高效、稳定、可预测，而不是理论上的绝对最优。

### 误区二：只要组件重新执行，DOM 就一定会完全重建

组件重新执行只说明 React 正在重新计算输出。若类型、位置和 `key` 保持稳定，React 会尽量复用既有节点，并只提交必要变更。

### 误区三：key 只是为了解决警告

`key` 的本质作用是节点身份标识，而不仅仅是消除控制台警告。它直接影响状态保留、节点复用和列表更新质量。

### 误区四：索引 key 总是可以接受

只有在列表完全静态、不会插入、删除、排序且没有局部状态时，索引 key 才勉强可接受。只要列表项存在身份、顺序变化或内部状态，索引 key 都可能造成隐藏问题。

### 误区五：Diff 只和列表有关

列表是 Diff 最典型的场景，但 Diff 并不局限于数组渲染。任何一次节点类型变化、子树替换、组件切换、条件分支切换，本质上都在触发 React 的比较与复用规则。

## 工程实践建议

如果把 React Diff 落到实际项目中，可以总结为几条高价值原则：

- 先从“节点身份”理解状态，而不是只从“组件是否重新渲染”理解状态
- 列表项尽量使用业务稳定 ID 作为 `key`
- 不要在 render 中生成随机 key
- 需要刻意重置状态时，可以主动更换 `key`
- 需要保留状态时，尽量保持类型、位置和 `key` 稳定
- 分析性能问题时，要区分“render 中比较成本”和“commit 中 DOM 提交成本”
