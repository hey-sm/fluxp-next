# Next.js 性能与工程组织

Next.js 项目的性能问题，很少是由单一 API 选择造成的。真正决定长期表现的，往往是几类结构性决策：服务端与客户端边界怎么划，数据是否被放在合适的位置，缓存是否按业务粒度设计，路由和组件是否过度耦合，静态资源与第三方脚本是否进入了错误链路。

这篇文章不做零散“性能技巧清单”，而是从工程组织层面收束整个系列，给出一套更稳定的优化判断框架。

- [Caching and Revalidating](https://nextjs.org/docs/app/getting-started/caching-and-revalidating)

- [Fonts](https://nextjs.org/docs/app/getting-started/fonts)

## 先建立一个核心认知

App Router 项目的性能优化，大体可以归为四层：

- 组件边界：哪些代码留在服务端，哪些进入客户端
- 数据边界：哪些请求缓存，哪些动态
- 路由边界：哪些段共享布局，哪些需要分段加载
- 资源边界：图片、字体、脚本和样式如何接入

真正有效的优化，通常都在这些边界上发生。

## 服务端优先

一句话定义：在 App Router 中，性能优化的第一原则通常是服务端优先，而不是客户端优先。

它解决什么问题：浏览器包体、Hydration 成本和交互前等待往往来自过度客户端化。

什么时候用：

- 页面主体内容
- 静态或弱交互内容
- 需要靠近服务端数据源的模块

常见注意点：把一整页随手标成 `'use client'`，通常比少写几个 Hook 带来的代价大得多。

## 客户端包体控制

一句话定义：客户端包体控制的核心，是只让真正需要浏览器执行的代码进入客户端边界。

它解决什么问题：包体越大，下载、解析和执行成本越高，交互就绪时间也越慢。

什么时候需要特别关注：

- 大型组件库接入
- 富文本、图表、地图、编辑器
- 把 layout 整体客户端化的场景

更稳的做法通常是：

- 客户端边界尽量小
- 重交互组件单独拆分
- 只在必要位置使用 `'use client'`

## 数据获取位置

一句话定义：把数据放在正确的位置获取，本身就是性能优化的一部分。

它解决什么问题：如果本可在服务端完成的读取被延后到客户端，页面会多一轮请求与等待；如果本可并行的请求被写成串行，也会拉长总耗时。

更稳的做法通常是：

- 首次内容优先服务端获取
- 能并行的请求尽量并行
- 慢模块单独切 Suspense 边界

## 缓存策略

一句话定义：缓存策略决定了页面是更偏“新鲜读取”还是更偏“可复用输出”，而不是单纯的性能开关。

它解决什么问题：很多页面性能差，不是因为 Next.js 不会缓存，而是因为缓存粒度与业务边界不匹配。

更稳的做法通常是：

- 先判断页面是否必须动态
- 数据可共享时用 tag 建模
- 变更后按路径或标签做精确重验证

常见注意点：粗暴地“全 no-store”会放弃大量框架优势，粗暴地“全 force-cache”则会牺牲新鲜度。

## 路由级拆分

一句话定义：路由级拆分指通过 route segment、layout、`loading.tsx` 和 Suspense，把页面等待和渲染责任分层。

它解决什么问题：复杂页面如果没有边界，会形成整页阻塞和过度重渲染。

更稳的做法通常是：

- 把共享外壳提到 layout
- 把慢块交给 Suspense
- 让 `loading.tsx` 只负责段级反馈

这样不仅更快，也更容易让问题定位回具体段。

## 资源接入

一句话定义：图片、字体和脚本的接入方式会直接决定布局稳定性、加载顺序和交互阻塞。

它解决什么问题：不少性能损耗不是页面逻辑本身，而是资源被错误地原始接入。

更稳的做法通常是：

- 图片优先 `next/image`
- 字体优先 `next/font`
- 第三方脚本优先 `next/script`

## 目录与模块组织

一句话定义：工程组织本身会影响性能，因为边界混乱的目录结构往往也意味着边界混乱的组件和数据流。

它解决什么问题：如果局部逻辑和全局逻辑不分，客户端和服务端模块混放，请求层和 UI 层耦合，项目很快会失去优化抓手。

更稳的做法通常是：

- 路由段专属逻辑 colocate
- 跨页面共享逻辑再抽离
- UI 共享层、服务层、领域模型层分开
- 客户端模块和服务端模块边界清晰

## 一个更稳的优化顺序

在真实项目里，更稳的优化顺序通常是：

1. 先检查是否过度使用 `'use client'`
2. 再检查数据请求是否位置错误或串行过多
3. 再检查缓存策略是否失配
4. 再检查慢块是否缺少 Suspense / `loading.tsx` 边界
5. 最后检查图片、脚本、字体与样式入口

这个顺序很重要，因为很多表面性能问题，根因其实是结构问题。

## 如何判断优化是否有效

一个更务实的判断方式通常是：

- 页面首屏是否更快出现稳定结构
- 导航是否减少了整页等待感
- 客户端交互是否更早可用
- 数据更新后是否仍保持正确新鲜度

也就是说，优化目标不只是 Lighthouse 分数，而是渲染链路和用户感知一起更合理。

## 常见误区

### 误区一：Next.js 自带优化，所以架构怎么写都不会太差

框架能提供能力边界，但真正能否吃到收益，取决于你是否顺着这些边界组织应用。

### 误区二：性能优化就是少写几个客户端 Hook

客户端 Hook 只是表象，更深层的问题往往是边界划分和数据链路位置。

### 误区三：只要加缓存，性能就会好

缓存如果与业务边界不匹配，只会把旧数据问题引进来。

### 误区四：工程组织只影响可维护性，不影响性能

在 App Router 中，这两者高度相关，因为边界组织本身就决定了渲染与打包边界。

## 文末速查

| 问题               | 优先检查的方向                     |
| ------------------ | ---------------------------------- |
| 页面包体过重       | `'use client'` 边界是否过大        |
| 首屏等待过长       | 数据位置、串行请求、Suspense 边界  |
| 导航切换不够快     | 预取、Router Cache、段级拆分       |
| 数据显示不新鲜     | 缓存策略与重验证                   |
| 图片和字体影响加载 | `next/image`、`next/font` 接入方式 |
| 项目越来越难优化   | 路由、模块和服务层组织是否混乱     |

如果只保留一个结论，可以记住下面这句话：

> Next.js 的性能优化，本质上是边界优化：组件边界、数据边界、路由边界和资源边界都要同时正确。
