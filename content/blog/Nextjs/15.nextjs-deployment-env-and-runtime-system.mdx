# Next.js 部署与运行时

Next.js 的运行方式并不只有“本地 `next dev` 能跑起来”这么简单。到了生产环境，应用会立刻面临一组工程问题：代码运行在 Node.js 还是 Edge，环境变量如何注入，哪些逻辑能进浏览器，哪些只能留在服务端，部署到 Vercel 和自托管时又有哪些边界差异。

这篇文章的目标，是建立从本地开发到生产运行的一套稳定认知，让运行时、环境变量和部署方式不再是零散配置，而是一条清晰链路。

官方文档主线参考：

- [Deploying](https://nextjs.org/docs/app/guides/deploying)
- [Self-Hosting](https://nextjs.org/docs/app/guides/self-hosting)
- [Environment Variables](https://nextjs.org/docs/app/guides/environment-variables)
- [Route Segment Config](https://nextjs.org/docs/app/api-reference/file-conventions/route-segment-config)

## 先建立一个整体认知

部署与运行时可以分成三层：

- 运行环境：Node.js Runtime、Edge Runtime
- 配置注入：环境变量与构建期 / 运行期边界
- 部署目标：Vercel、容器化、自托管

很多生产问题，本质上都是这三层边界没分清。

## Node.js Runtime

一句话定义：Node.js Runtime 是 Next.js 服务端逻辑最常见的运行环境，适合大多数服务端组件、Route Handlers 和数据访问场景。

它解决什么问题：多数真实业务依赖 Node.js 生态、数据库客户端、文件系统或完整服务端库，这些能力通常更适合放在 Node.js 运行时。

什么时候用：

- 数据库访问
- Node.js SDK
- 大多数页面和接口逻辑

常见注意点：如果没有明确理由，不必主动追求 Edge。Node.js Runtime 才是多数 Next.js 项目的默认主线。

## Edge Runtime

一句话定义：Edge Runtime 适合对延迟敏感、逻辑轻量、依赖受限的边缘执行场景。

它解决什么问题：某些场景希望在更靠近用户的位置执行轻量逻辑，以降低延迟。

什么时候用：

- 地理位置敏感的轻量路由逻辑
- 边缘个性化分流
- 某些低延迟前置处理

常见注意点：Edge 不是“更先进的默认选择”。它有更严格的运行时限制，不适合依赖 Node.js 全生态的业务逻辑。

## runtime 配置

一句话定义：Next.js 可以通过 route segment config 指定部分逻辑使用特定运行时。

它解决什么问题：不同页面或接口对运行环境的需求可能不同，需要显式声明。

```ts
export const runtime = 'nodejs'
```

常见注意点：运行时选择不是性能开关，而是能力边界声明。选错运行时，最先暴露的通常不是速度，而是依赖不可用。

## 环境变量

一句话定义：环境变量用于把部署环境相关配置从代码中抽离出来，并在不同环境下注入不同值。

它解决什么问题：API 地址、密钥、数据库连接、实验开关等都不应硬编码进仓库。

什么时候用：

- 服务端密钥
- 第三方服务配置
- 构建环境和部署环境差异

```env
DATABASE_URL=postgres://...
NEXT_PUBLIC_API_BASE=https://example.com
```

常见注意点：只有以 `NEXT_PUBLIC_` 开头的环境变量会暴露到客户端。其余变量默认只应在服务端使用。来源：[Environment Variables](https://nextjs.org/docs/app/guides/environment-variables)

## 构建期与运行期边界

一句话定义：某些配置在构建期就会被确定，另一些配置则在运行期读取。

它解决什么问题：很多部署问题并不是变量没配，而是把“构建时决定的值”误当成“上线后还能热切换的值”。

常见注意点：尤其在客户端暴露变量时，要明确它们会进入构建产物，不应把会频繁变动且敏感的值放入客户端公开变量。

## 部署到 Vercel

一句话定义：Vercel 是 Next.js 的默认托管语境，许多优化路径与平台能力天然协同。

它解决什么问题：对大多数团队而言，Vercel 可以最直接承载 App Router、缓存、边缘执行和静态资源分发。

什么时候用：

- 希望最少平台适配成本
- 以 Next.js 为核心应用栈

常见注意点：虽然 Vercel 是自然语境，但并不意味着所有运行时和缓存行为都只能在 Vercel 下理解；只是部署体验更顺。

## 自托管

一句话定义：Next.js 也支持自托管，但需要你自己承担更多运行、缓存、进程和静态资源分发层面的责任。

它解决什么问题：某些团队出于合规、基础设施统一或成本考虑，会选择容器或自有平台部署。

什么时候用：

- 企业内部平台
- 合规要求
- 与现有平台统一部署

常见注意点：自托管不是“构建完扔到 Node 进程里”就结束，缓存、静态资源、进程模型和观测能力都需要单独考虑。来源：[Self-Hosting](https://nextjs.org/docs/app/guides/self-hosting)

## 环境变量的稳定使用顺序

更稳的做法通常是：

- 服务端变量默认不暴露给客户端
- 客户端真正需要的才使用 `NEXT_PUBLIC_`
- 平台差异配置优先通过环境变量而不是硬编码分支
- 在本地、预览、生产环境分别管理注入

这条顺序的重点，是把“敏感性”和“可变性”同时纳入考虑。

## 一个更稳的部署判断顺序

在真实项目里，更稳的判断顺序通常是：

- 先确认服务端逻辑是否依赖 Node.js 生态
- 大多数页面和接口默认使用 Node.js Runtime
- 只有确实需要边缘前置或低延迟能力时再考虑 Edge
- 优先把环境变量边界理清，再进入平台部署
- 需要自托管时，提前补齐缓存、日志和运维认知

这样能避免把部署决策简化成“平台品牌选择”。

## 常见误区

### 误区一：Edge Runtime 一定比 Node.js Runtime 更好

不准确。Edge 是场景化选择，不是普适升级。

### 误区二：只要变量写进 `.env` 就会自动安全

变量是否安全，取决于它是否暴露给客户端、是否进入构建产物，以及平台如何注入。

### 误区三：Next.js 只能在 Vercel 上好用

Vercel 是自然语境，但并不是唯一部署方式；只是自托管需要更多工程承担。

### 误区四：部署问题只和平台相关，和代码结构无关

运行时边界、环境变量使用方式、缓存设计都会直接影响部署复杂度。

## 文末速查

| 场景               | 优先想到的能力  |
| ------------------ | --------------- |
| 大多数服务端逻辑   | Node.js Runtime |
| 轻量低延迟边缘逻辑 | Edge Runtime    |
| 指定路由运行环境   | `runtime` 配置  |
| 服务端密钥         | 普通环境变量    |
| 客户端公开配置     | `NEXT_PUBLIC_`  |
| 最少平台适配成本   | Vercel          |
| 平台自主可控       | 自托管          |

如果只保留一个结论，可以记住下面这句话：

> Next.js 的部署问题，本质上是在回答“代码在哪运行、配置何时注入、平台由谁负责”。
